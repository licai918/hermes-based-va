# Workbench login cutover to Postgres via server-side authenticate

Increment 5 of the local-first migration (#38) cuts the **login / session path** over to the Postgres system-of-record (ADR-0140) so account management *and* login share one store. Increment 4 moved account CRUD onto the Supervisor Admin per-profile API (ADR-0141) but left login reading the in-memory `AccountStore` (`auth-handlers.ts` → `deps.store.getByUsername`). That split-brain (I-1) meant API-created accounts could not log in, and in-memory seed accounts logged in but never appeared in the admin list. It also left `last_login_at` unbacked (M-1), so the admin list's "last login" was always null.

`toee_workbench_admin` gains a fifth v1 **Domain Adapter Tool Action**, `authenticate`, extending ADR-0069. The in-memory login path stays the behavioral source of truth: the API path must return the same responses for valid creds / bad password / unknown user / disabled account, and issue the session identically.

## Decision

- **Password verification happens server-side in Hermes/Postgres**, never in the BFF. `toee_workbench_admin.authenticate` takes `{username, password}` (plaintext), reads the stored `password_hash`, verifies it with the scrypt KDF, and on success returns the **public account only** (`_account_row`: id/username/role/status/created_at/updated_at/last_login_at — **never the hash**). The hash never leaves the Hermes server. This is strictly more secure than verifying in the BFF, which would require the hash to cross the wire *and* sit in BFF memory; here the hash stays pinned to the datastore.
- **Plaintext on the wire is bounded to the ADR-0141 trust boundary.** The plaintext password transits only the internal, bearer-authed, localhost dispatch channel (BFF → admin dispatch server), the same channel over which `create_account` already sends a derived `password_hash`. Local-first (ADR-0142) keeps it on one host; deployed envs MUST run this channel over TLS (server-to-server within the project, Cloud Run packaging deferred to Slice 37). This is not a new trust assumption — it is ADR-0141's shared-bearer model, restated for a pre-auth call.
- **`authenticate` is pre-auth and fail-open on actor.** No acting account exists yet — `authenticate` *establishes* the actor — so the handler does **not** call `_require_actor`; the BFF dispatches it with `client.dispatch` (read-style), not `dispatchWrite`. It is a read-style verification, not a governed mutation, with one write side effect: on success it updates `last_login_at` in the same transaction. That write is intentionally not actor-gated because the subject *is* the authenticating user (resolves M-1). Login is recorded by `last_login_at`, not by a separate Workbench Audit Log row — parity with the in-memory path, which audits no login.
- **Profile availability needs no allowlist change.** `authenticate` rides `toee_workbench_admin`, already allowlisted for the **Supervisor Admin Profile** (ADR-0038). The Profile Tool Allowlist gates by toolset (ADR-0034), so adding an action exposes it to that profile automatically — the same reasoning as ADR-0143. Login authenticates against the same Postgres accounts the admin API manages, so it uses the **admin** profile API (`HERMES_ADMIN_API_URL`/`HERMES_ADMIN_API_TOKEN`).
- **Parity & no user-enumeration.** The in-memory path returns the *same* generic `401 invalid credentials` for unknown user and bad password. `authenticate` raises one governed class — `unauthenticated` (new; BFF → 401) — for **both**, with an identical message, so neither status nor body leaks which. A disabled account is checked **before** the password (matching in-memory order) and raises `policy_blocked` (BFF → 403 `account disabled`), matching in-memory behavior exactly — including that this is distinguishable from the 401 (pre-existing behavior, preserved for parity, not introduced here).
- **BFF login route.** When the admin API is configured, `handleLoginViaApi` dispatches `authenticate`, maps the result with the existing `mapPublicAccount` whitelist, issues the session exactly as `handleLogin` does (same `{ user: { accountId, username, role } }` body + signed cookie), and on `unauthenticated` returns the byte-identical `problem(401, "invalid credentials")`. Otherwise it keeps the in-memory path. Session issuance/validation stays BFF-side and unchanged (ADR-0093). Plaintext and hash are never logged or returned.

## Consequences

- The catalog (`@toee/shared` + `toee_hermes`), both deterministic mock stubs, and the plugin manifest all carry `toee_workbench_admin__authenticate`; the Postgres datastore implements the real verify + `last_login_at` write. A new `0002_account_last_login.sql` migration adds the nullable `last_login_at TIMESTAMPTZ` column (the live DB already applied `0001`, so this must be a new migration), and `_account_row`/`_list_accounts` surface it so the admin list shows real last-login.
- **Lockout gap (M-2) — RESOLVED in 0.0.4 S08.** As originally written: ADR-0018 brute-force lockout (`failed_attempts`/`locked_until`, the in-memory `423` path) was **not** carried to the API path in this increment — Postgres had no failed-attempt state, so with the admin API configured, repeated bad passwords were not throttled. The named upgrade path was taken exactly as written: migration `0013_account_lockout` adds the two columns, and `authenticate` now does check-before-verify / increment-on-fail / reset-on-success. A `locked` error class carries the verdict (the governed message is replaced upstream, so the class is the only signal that survives dispatch) and the BFF maps it to `423`. The ladder is now durable across a restart, which the in-memory store never was.
- **Mock backend is not an auth authority.** Like every mock stub (ADR-0137), `authenticate`'s mock returns a deterministic success shape and verifies nothing. A real login surface MUST point at an admin dispatch server running `TOOL_BACKEND=datastore`; `TOOL_BACKEND=mock` treats every credential as valid and is dev/test only.
- Cross-runtime hash compatibility is load-bearing: the TS `hashPassword` (`scrypt$saltHex$hashHex`, Node `scryptSync` defaults N=16384/r=8/p=1, 64-byte key) is verified by Python `hashlib.scrypt` with the same parameters. A datastore test pins a TS-generated hash to guard the two implementations against drift.

**Considered options:** verify the password in the BFF after fetching the hash over the API (rejected — puts the hash on the wire and in BFF memory; server-side verify keeps the hash in the datastore); reuse a `not_found`/`policy_blocked` class for bad credentials instead of adding `unauthenticated` (rejected — neither maps to 401, and `not_found` for unknown-user would leak enumeration); return a structured `{ ok: false, reason }` success-shape instead of a governed error (rejected — risks the BFF treating a failure as a login success; a governed raise rolls back the `last_login_at` touch and fails closed); carry full ADR-0018 lockout in this increment (deferred — see M-2; out of the enumerated parity scope and a meaningful column/logic addition of its own).
