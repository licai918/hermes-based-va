# Workbench knowledge-slots cutover to Postgres via a dedicated authoring table

Increment 6 of the local-first migration (#43, remainder of Slice 35 #38) cuts the **Supervisor Admin KnowledgeOps slot routes** (`/admin/knowledge`, ADR-0087) over to the Postgres system-of-record (ADR-0140). Increments 4ā€“5 moved account CRUD and login onto the Supervisor Admin per-profile API (ADR-0141/0144); knowledge slots were **stop-and-reported on #43** because ā€” unlike accounts ā€” the existing datastore handler (`knowledge.py` over `knowledge_version`) modelled a *different shape* than the in-memory `KnowledgeStore`, which is the behavioral source of truth. Reaching parity needed a schema (re)design, deferred so it could be vetted rather than silently expanded (cf. the drafts/Textline deferral #41/#42).

The in-memory `KnowledgeStore` (`apps/workbench/lib/gateway/knowledge-store.ts`) is the contract this cutover must reproduce. It supports exactly four operations over six fixed slots, returning the `PolicySlot` read model `{ slotId, title, status, draftText, publishedText, owner, reviewDate, hasGapPrompt }`:

- `listSlots()` ā†’ `GET /api/admin/knowledge/slots`
- `saveDraft(slotId, { draftText?, owner?, reviewDate? })` ā†’ `PUT /api/admin/knowledge/slots/[id]`
- `submitForEval(slotId)` ā†’ `POST .../[id]/submit`
- `rollbackPublished(slotId)` ā†’ `POST .../[id]/rollback`

## Why the existing `knowledge_version` model could not be reused as-is

`knowledge_version` (ADR-0003/0040, the eval **publish-gate** model) diverges from the store on every axis the UI depends on:

| Concern | In-memory `KnowledgeStore` (source of truth) | `knowledge_version` |
|---|---|---|
| Slot identifier | kebab UI id (`business-hours`) | ADR-0003 snake key (`business_hours_service_boundaries`) |
| Text | **separate** `draftText` + `publishedText` on one slot | single `content` column per version row |
| Metadata | `owner`, `reviewDate`, `hasGapPrompt` | none |
| Rollback | restores the **previous published text** from history | marks the published row `rolled_back`; keeps no prior text to restore |
| `gap` status | `gap` + `hasGapPrompt` exist | not modelled |

These are not snakeā†”camel renames; they are different identifiers, a different cardinality (draft *and* published coexist on one slot), and missing fields. Forcing the store shape onto `knowledge_version` would have been a larger, riskier diff than a purpose-built table.

## Decision

- **A new dedicated authoring table `workbench_policy_slot` (+ `workbench_policy_slot_history`)** is the Postgres home for the KnowledgeOps slot feature, modelled to mirror `PolicySlot` field-for-field (kebab `slot_id` PK, `title`, `status`, `draft_text`, `published_text`, `owner`, `review_date`, `has_gap_prompt`, plus a `sort_order` for the fixed ADR-0003 list order and `created_at`/`updated_at`). Rollback's "previous published version" is a `workbench_policy_slot_history` stack (`slot_id`, `published_text`, `archived_at`), popped most-recent-first.
- **`knowledge_version` + `publish_pending_slot` + `toee_eval_review.promote_pending_policy` are left UNTOUCHED.** They are the eval publish-gate subsystem and are #44 territory (deferred). The four `toee_knowledge_ops` *authoring* handlers are reimplemented against the new table; the catalog action names, the plugin manifest, the Supervisor-Admin allowlist, and both mock stubs are **unchanged** (`toee_knowledge_ops` and its four actions already exist ā€” this increment only swaps what the datastore handlers do, exactly as accounts/login added behavior to existing tools).
- **The six Required Operational Policy Slots are seeded as empty onboarding placeholders** (ADR-0003: structured-but-unfilled from onboarding) by the `0003` migration ā€” kebab ids + store titles, `status = 'empty'`. The in-memory **demo seed** (pre-published `business-hours` with prior-version history, `gap` `exception-scripts`, etc.) is intentionally **not** carried into Postgres, matching the accounts cutover precedent that a real DB starts empty. Seeding the rows (vs. lazy-creating) means an untouched-but-known slot returns `no draft to submit` (conflict) rather than `not found`, preserving store parity.
- **Writes are fail-closed on actor + audited (ADR-0141/0029).** `saveDraft`/`submit`/`rollback` call `_require_actor` (no attributed actor ā†’ governed `policy_blocked` ā†’ 403, no mutation, no NULL-actor audit row) and append a `workbench_audit_log` row (`update_policy_slot` / `submit_for_eval` / `rollback_published_policy`, `target_type = policy_slot`, `target_id = slotId`) in the same transaction. The BFF dispatches reads with `client.dispatch` (fail-open) and writes with `client.dispatchWrite` (fail-closed). This is the governance the in-memory store lacked ā€” the same addition accounts made.
- **Atomic SQL for the conflict/not-found classes (ADR-0104 status parity):**
  - `update_policy_slot` is one `UPDATE ā€¦ WHERE slot_id = %s RETURNING ā€¦`; `rowcount 0` ā†’ `not_found` (404). It applies only the provided fields (`COALESCE(%(field)s, col)`) and flips `empty`/`gap` ā†’ `draft` when the resulting `draft_text` is non-empty, in SQL ā€” faithful to the store's `saveDraft`.
  - `submit_for_eval` is `UPDATE ā€¦ WHERE slot_id = %s AND draft_text <> '' RETURNING ā€¦`; on `rowcount 0` it disambiguates with one existence probe ā†’ `not_found` (404, unknown slot) vs. `conflict` (409, "slot has no draft to submit").
  - `rollback_published_policy` probes existence (ā†’ 404), then atomically pops history with `DELETE ā€¦ WHERE id = (SELECT ā€¦ ORDER BY archived_at DESC LIMIT 1) RETURNING published_text`; no row ā†’ `conflict` (409, "slot has no previous published version"); else it writes `published_text`/`status='published'`. The single-statement pop means two concurrent rollbacks cannot claim the same history row.
- **BFF mapping + guards (ADR-0070).** `mapPolicySlot` (in `lib/bff/admin/knowledge.ts`, alongside the handlers ā€” mirroring `mapPublicAccount` in `accounts.ts`, not the case-focused `hermes-map.ts`) maps the snake_case slot row onto `PolicySlot`, rejecting an unknown `status` or missing `slot_id`/`title` as a governed `HermesApiError` so a bad upstream surfaces on the ADR-0090 banner. The four routes branch API-vs-store on `HERMES_ADMIN_API_URL`/`HERMES_ADMIN_API_TOKEN` (the admin profile, same env as accounts/login), preserving status parity (200/404/409) and the `{ slot }` / `{ slots }` body.

## Divergences from the in-memory shape ā€” review here (each is vetoable)

1. **Two slot-id namespaces now coexist (sign-off).** The authoring table keys on the kebab UI ids (`business-hours`); `knowledge_version` + `eval/policy_slot_map.yaml` + `operational_policy.py` key on ADR-0003 snake keys (`business_hours_service_boundaries`). This increment does **not** unify them ā€” it faithfully carries the store's ids. A future slice that connects authoring to the eval gate must map the two (or migrate one).
2. **Authoring is decoupled from the eval publish-gate (sign-off).** `submit_for_eval` flips the *authoring* slot to `pending_eval`, but nothing promotes it to `published` on `workbench_policy_slot` ā€” promotion still runs on `knowledge_version` via `promote_pending_policy`. This faithfully mirrors today's in-memory reality (the store has no publish path either; publish lives in the eval gate). **Consequence:** on a fresh Postgres DB nothing ever writes `published_text` or pushes history, so `rollback` returns `no previous published version` (409) until the publish path is connected (#44) or history is otherwise populated. Rollback is wired correctly; it simply has no data to act on yet. *If reviewers want authoringā†”publish connected now, that is the #44 redesign and should be scoped explicitly rather than bolted on here.*
3. **The demo seed is dropped (sign-off, low-risk).** The in-memory list shows `business-hours`/`payment-methods` as published with demo copy, `order-delivery` as draft, `accounting-inquiry` as pending_eval, `exception-scripts` as a gap. The Postgres list shows all six as empty placeholders. This is a **data** difference (not behavioral) and matches accounts (API mode starts with an empty account list). Reviewers who want a non-empty demo in Postgres should say so; shipping fake policy copy in a forward migration is deliberately avoided.
4. **Writes can now 403 (policy_blocked); the store never did.** Added governance, not a regression ā€” the BFF bakes the supervisor's `accountId` into the client, so 403 only fires on misconfiguration (defense-in-depth identical to accounts).
5. **`review_date` is stored as free-form `TEXT`, unvalidated.** The store accepts any string (e.g. `"2026-09-01"`) and never validates it; the table matches that rather than imposing a `DATE` type that would reject store-legal values.
6. **Error message text is governed-equivalent, not byte-identical.** Statuses match exactly (404/409); messages match the store's wording where set in the datastore (`"slot not found"`, `"slot has no draft to submit"`, `"slot has no previous published version"`), but the ViaApi body additionally carries `errorClass` (same as the accounts cutover via `hermesErrorToProblem`).

## Consequences

- New `0003_knowledge_slots.sql` (append-only; the live DB already applied `0001`/`0002`, so a new file, not an edit) creates the two tables, a history index, and the six placeholder rows. `migrate.py`'s ledger applies it; a migration test asserts the seed.
- `knowledge.py` keeps `publish_pending_slot`/`_latest_version`/`_next_version`/`_require_known_slot` for the (untouched) eval gate and gains `_require_actor` (mirroring `accounts._require_actor`). The previously-speculative `knowledge_version`-based authoring handlers are replaced; the datastore test that coupled `submit_for_eval` ā†’ `promote_pending_policy` is split (authoring lifecycle on the new table; promote seeded directly on `knowledge_version`, since the two are now decoupled per divergence #2).
- TDD on both sides: datastore (six seeded placeholders; save-draft flip + 404 + actor-less denial leaves no mutation/audit; submit pending/no-draft-conflict/not-found; rollback not-found/no-previous-conflict + a seeded-history restore), an e2e dispatch pair (actor ā†’ audit; actor-less ā†’ policy_blocked, no audit), the migration seed assertion, and BFF ViaApi (list mapping, dispatched params + actor on writes, per-class status, actor-less guard).

**Considered options:** overlay the store shape onto `knowledge_version` (rejected ā€” different id namespace, single-content vs. draft+published, and no owner/review/gap/history; a bigger, more fragile diff than a new table); a single `previous_published_text` column instead of a history table (rejected ā€” the store type is a stack; a forward-only migration makes the durable choice expensive to revisit, so model the stack now even though current data never exceeds one level); carry the in-memory demo seed into the migration (rejected ā€” ships fake policy copy; accounts precedent is empty-in-API-mode); unify the kebab/snake slot ids in this increment (deferred ā€” couples to the eval gate redesign, #44); connect authoring `submit` to the publish gate now (deferred ā€” #44, see divergence #2).
