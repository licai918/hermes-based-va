# Workbench eval-runs + publish-gate cutover to Postgres, with the authoring‚Üîpublish bridge

Increment 7 of the local-first migration (#44, the deferred remainder of Slice 35 #38) cuts the **Supervisor Admin Launch Eval Review routes** (`/admin/eval`, ADR-0088) over to the Postgres system-of-record (ADR-0140) and ‚Äî the centerpiece ‚Äî wires the **authoring‚Üípublish bridge** that ADR-0145 deliberately left open (its divergences #1 and #2). Increments 4‚Äì6 moved accounts (`c9f7d48`), login (`d8cb22b`), and KnowledgeOps slots (`81c1acd`, ADR-0145) onto the per-profile API (ADR-0141). Eval runs were **stop-and-reported on #44** because ‚Äî unlike accounts ‚Äî the `eval_run` model and `promote_pending_policy` semantics diverged from the in-memory `EvalStore`, the behavioral source of truth, and reconciling them needed a schema + promote-semantics redesign rather than a mechanical cutover.

The in-memory `EvalStore` (`apps/workbench/lib/gateway/eval-store.ts`) is the contract this cutover reproduces. Four operations over the standard ADR-0074 report:

- `listRuns()` ‚Üí `GET /api/admin/eval/runs` (most-recent-first, compact `EvalRunSummary`)
- `getRun(runId)` ‚Üí `GET .../[id]` (full `EvalRunReport`)
- `signOffMedium(runId)` ‚Üí `POST .../[id]/sign-off`
- `promotePending(runId)` ‚Üí `POST .../[id]/promote`

## Why the existing `eval_run` model + `promote_pending_policy` could not be reused as-is

| Concern | In-memory `EvalStore` (source of truth) | Pre-cutover datastore (`eval_run` + `eval_review.py`) |
|---|---|---|
| Governance state | `signed_off` **and** `promoted` are independent, **overlapping** booleans (a run can be both) | single `status` TEXT column ‚Äî cannot hold both |
| Severity projection | `summary.failed_high` **and** `summary.failed_medium` | only a `failed_high` column; no `failed_medium` |
| Read model | the full ADR-0074 report (`suite/model/prompt/knowledge/timestamp/scenarios/summary/signoff_required`) + the overlay | `id/suite/status/failed_high/report` projection that does not match the wire shape |
| Promote key | **runId-keyed**: marks the *run* promoted, gated by `suite == policy_publish` / `failed_high == 0` / sign-off | **slot-keyed**: published a `knowledge_version` row via `publish_pending_slot` ‚Äî a different action entirely |
| Not-found class | 404 | `unexpected_error` (‚Üí 502), not 404 |

These are not snake‚Üîcamel renames; the state model, the projection, and the promote contract all differ.

## Decision

- **`eval_run` gains the governance overlay as columns; the ADR-0074 report stays JSONB.** A new append-only migration (`0004_eval_run_governance.sql`) adds `signed_off BOOLEAN`, `promoted BOOLEAN` (the two independent overlapping flags the single `status` column could not represent), and `slot_key TEXT` (the publish bridge, below). The full report (`run_id/suite/model_slug/prompt_version/knowledge_version/timestamp/scenarios/summary{total,passed,failed_high,failed_medium}/signoff_required`) is the existing `report` JSONB ‚Äî exactly the EvalStore's split ("Governance overlay (not part of the on-disk report; merged in by the store)"). Handlers project the report from JSONB and overlay `signed_off`/`promoted` from the columns, so `failed_medium` and `signoff_required` are read from the report, not duplicated as columns. The legacy `suite`/`status`/`failed_high` columns remain (append-only) but are not the projection's source of truth.
- **`promote_pending_policy` is reworked to be runId-keyed with EvalStore-parity gating** (`run_id` param): not_found ‚Üí 404; `suite != policy_publish` ‚Üí conflict "run is not a promotable policy_publish run"; `failed_high > 0` ‚Üí conflict "high-severity failures block promotion"; `signoff_required && !signed_off` ‚Üí conflict "medium failures must be signed off first"; else mark `promoted = TRUE`. `sign_off_medium_failure` matches the store likewise (not_found / "high-severity failures block sign-off" / "no medium sign-off required"). Both are **idempotent** like the store's `Set.add`: re-promote of an already-promoted run is an early success that does **not** re-publish (no double history push); re-sign-off succeeds again (it has no side effect to guard).
- **The publish bridge (resolves ADR-0145 divergence #2).** Promotion of a `policy_publish` run that carries a `slot_key` now **publishes the matching authoring slot** on `workbench_policy_slot` (ADR-0145): it pushes the slot's current `published_text` (if any) onto `workbench_policy_slot_history`, then sets `published_text = draft_text` and `status = 'published'`. This is the bridge ADR-0145 flagged: `submit_for_eval` flips the kebab authoring slot to `pending_eval`, and **promotion is now the publish step**, so on a real submit‚Üírecord‚Üípromote flow `rollback_published_policy` has history to restore ‚Äî closing the "fresh DB always 409s" gap. The push + publish + the run's `promoted` flag are one statement-sequence inside the driver's **single transaction** (`datastore/driver.py` commits on return, rolls back on raise), so a mid-publish failure leaves neither a half-promoted run nor an orphaned history row.
- **The id-namespace bridge (resolves ADR-0145 divergence #1).** Authoring keys on kebab UI ids (`business-hours`); the eval gate keys on ADR-0003 snake keys (`business_hours_service_boundaries`, shared with `policy_slot_map.yaml`/`REQUIRED_POLICY_SLOTS`). `eval_run.slot_key` holds the **snake** key (the eval-gate namespace a `policy_publish` run belongs to); the bridge maps it to the kebab authoring id through one **explicit, total, bijective** map ‚Äî `POLICY_SLOT_ID_BY_KEY` in `operational_policy.py`, co-located with `REQUIRED_POLICY_SLOTS` and bound by a test so the two cannot drift. The publish is then a **direct primary-key** `UPDATE ‚Ä¶ WHERE slot_id = %s`, so a cross-namespace lookup can never hit the wrong row; an unknown `slot_key` is refused (`unexpected_error`) rather than guessed.
- **`knowledge_version` + `publish_pending_slot` + `_latest_version` + `_require_known_slot` are removed** (deletion over addition): with promote no longer publishing a `knowledge_version` row, they were dead. The `knowledge_version` *table* is left in place (append-only migration; an empty, now-unused table) ‚Äî publishing onto it was the divergent datastore-only path the EvalStore never had; the authoring table (ADR-0145) is the real published-policy home.
- **Writes are fail-closed on actor + audited (ADR-0141/0029).** `sign_off_medium_failure`/`promote_pending_policy` call `_require_actor` (no attributed actor ‚Üí governed `policy_blocked` ‚Üí 403, no mutation, no NULL-actor audit row) and append one `workbench_audit_log` row (`sign_off_medium_failure` / `promote_pending_policy`, `target_type = eval_run`, `target_id = runId`) in the same transaction; the bridge publish records the published `slot_key` in that promote row's `details` rather than a second audit. Reads (`list`/`get`) dispatch fail-open. This is the governance the in-memory store lacked ‚Äî the same addition accounts/knowledge made.
- **Atomic SQL for the governed classes (ADR-0104 status parity).** Sign-off is one `UPDATE ‚Ä¶ WHERE id = %s AND (report#>>'{summary,failed_high}')::int = 0 AND (report->>'signoff_required')::boolean RETURNING ‚Ä¶`; on `rowcount 0` one existence/gate probe disambiguates not_found (404) vs the two 409 messages. Promote is `UPDATE ‚Ä¶ WHERE id = %s AND <suite/failed_high/sign-off gates> AND NOT promoted RETURNING ‚Ä¶`; `rowcount 0` disambiguates not_found / not_promotable / failed_high / signoff_required, **or** an already-`promoted` run (idempotent success, no re-publish). The single-statement gate means a concurrent sign-off cannot race the promote check.
- **BFF mapping + guards (ADR-0070).** `mapEvalRunSummary` + `mapEvalRunReport` (in `lib/bff/admin/eval.ts`, beside the handlers ‚Äî mirroring `mapPolicySlot`/`mapPublicAccount`) validate the dispatched rows (required ids, numeric summary, known scenario severities, boolean overlay) and reject a contract violation as a governed `HermesApiError` so a bad upstream surfaces on the ADR-0090 banner. The four routes branch API-vs-store on `HERMES_ADMIN_API_URL`/`HERMES_ADMIN_API_TOKEN` (the admin profile, same env as accounts/login/knowledge), preserving status parity (200/404/409) and the `{ run }` / `{ runs }` body.

## Divergences from the in-memory shape ‚Äî review here (each is vetoable)

1. **The publish bridge is genuinely new product behavior, not a translation (sign-off ‚Äî the key decision).** The in-memory `EvalStore.promotePending` marks a run promoted and touches **no** policy state; it never published anything. ADR-0145 flagged (divergence #2) that authoring `submit` and the publish gate were decoupled, and #44 was explicitly scoped to connect them. This increment makes **promotion the publish step** for an authoring slot. The behavior is specified (ADR-0145 divergence #2, and the #44 instruction), but it is invented here, so it is the most important thing to veto if the intended product flow differs (e.g. publish should be its own action, or should clear `draft_text`, or should require the slot to be `pending_eval`).
2. **`eval_run.slot_key` links a run to the slot it gates (sign-off).** The in-memory run has no slot reference, so this column is new. A `policy_publish` run with no `slot_key` promotes (run marked promoted) but publishes nothing ‚Äî faithfully matching "promote just marks the run" for unlinked runs, and engaging the bridge only when the link exists. Who sets `slot_key` at record time (the eval runner, from `policy_slot_map.yaml`) is a future seam; today it is set by the recorder/seed. Veto if runs should instead carry the kebab id directly, or if the link should be a FK.
3. **The kebab‚Üîsnake correspondence is asserted by `POLICY_SLOT_ID_BY_KEY`, not derived (sign-off, low-risk).** The six-entry bijection is hand-written and test-bound to `REQUIRED_POLICY_SLOTS` (snake) and the authoring seed (kebab). It is the single reconciliation point. Veto if the two namespaces should be unified (migrate one) rather than bridged.
4. **`knowledge_version` publishing is retired (sign-off, low-risk).** Promote no longer writes `knowledge_version`; the table is now an unused stub. No production reader consumed it (knowledge search is a separate stub seam). Veto if a future reader is expected to depend on the `knowledge_version` lifecycle.
5. **Writes can now 403 (`policy_blocked`); the store never did.** Added governance, not a regression ‚Äî the BFF bakes the supervisor's `accountId` into the client, so 403 only fires on misconfiguration (identical to accounts/knowledge).
6. **Error message text is governed-equivalent, not byte-identical.** Statuses match exactly (404/409); messages match the store's wording set in the datastore ("run not found", "no medium sign-off required", "high-severity failures block sign-off", "run is not a promotable policy_publish run", "high-severity failures block promotion", "medium failures must be signed off first"); the ViaApi body additionally carries `errorClass` (same as accounts/knowledge).
7. **No demo seed in Postgres (sign-off, low-risk).** `createEvalSeed()`'s three demo runs are not carried into the migration ‚Äî like accounts/knowledge, a real DB starts empty; the eval runner records real runs. The list is empty until then.

## Consequences

- New `0004_eval_run_governance.sql` (append-only; the live DB already applied `0001`‚Äì`0003`) alters `eval_run`. `migrate.py`'s ledger applies it; a migration test asserts the three new columns.
- `eval_review.py` is reworked: `list`/`get` project the report+overlay (get raises `not_found`); `sign_off_medium_failure`/`promote_pending_policy` become fail-closed governed writes with EvalStore-parity gating; promote runs the publish bridge. `knowledge.py` loses the dead `knowledge_version` helpers; `eval_review.py` loses its `publish_pending_slot` import. `operational_policy.py` gains `POLICY_SLOT_ID_BY_KEY` + `authoring_slot_id_for`.
- TDD on both sides: datastore (list ordering + projection; get 404; sign-off pass/failed_high/not_required/not_found/actor-less denial; promote not_promotable/failed_high/signoff-then-promote/not_found/actor-less denial/idempotent; **the end-to-end submit‚Üírecord‚Üípromote‚Üípublish‚Üírollback flow ADR-0145 could not test**; the kebab‚Üîsnake bijection; the migration columns), an e2e dispatch pair (admin actor ‚Üí audit; actor-less ‚Üí policy_blocked, no audit), and BFF ViaApi (summary/report mapping, dispatched params + actor on writes, per-class status, actor-less guard).
- The tool catalog, `plugin.yaml`, the Supervisor-Admin allowlist, and both mock stubs are **unchanged** ‚Äî `toee_eval_review` and its four actions already exist; this increment only changes what the datastore handlers do, exactly as accounts/login/knowledge did.

**Considered options:** keep promote slot-keyed and add a separate runId-keyed action (rejected ‚Äî #44 says promote must match the store's runId key; two promote actions is more surface, not less); promote the report fields to first-class columns instead of projecting JSONB (rejected ‚Äî the report is an ADR-0074 JSON document; columns would duplicate it and ossify the schema against report evolution, for no query the admin console makes); store the kebab id on `eval_run` and skip the bijection (rejected ‚Äî the eval gate is the snake namespace; a kebab id there leaks the authoring namespace into the gate and still needs the map to validate); unify the kebab/snake ids in this increment (deferred ‚Äî a data migration of one namespace is larger than the bridge and not required for the flow); leave `knowledge_version` publishing in place beside the new bridge (rejected ‚Äî two publish targets is the ambiguity #44 exists to remove).
