## Implementer choices (stated per the issue's "implementer's choice" calls)

- **Module layout**: `hermes_runtime/knowledge/` package (`config.py`, `migrate.py`),
  mirroring `hermes_runtime/datastore/` 1:1 in structure and API shape
  (`knowledge_database_url()`, `discover_migrations()`, `run_migrations()`,
  `migrate()`, `main()`).
- **Migrations dir**: separate directory `hermes-runtime/knowledge_migrations/`
  with its **own** `0001`-based numbering (not a shared sequence with
  `hermes-runtime/migrations/`). Each knowledge-DB connection gets its own
  `schema_migrations` bookkeeping table — no collision risk since it's a
  different Postgres database entirely (`toee_knowledge` vs `toee_va`).
- **Embedding storage**: `BYTEA` column (`embedding`), nullable, unused this
  slice. No pgvector extension dependency. Documented in the migration's SQL
  comment as the explicit decision + revisit trigger.
- **id**: `BIGSERIAL` (DB-generated identity), not app-generated — matches the
  spike's `ingest.py`, which does `TRUNCATE ... RESTART IDENTITY` and never
  supplies an id itself.

## Schema (`hermes-runtime/knowledge_migrations/0001_knowledge_chunk.sql`)

```sql
CREATE TABLE knowledge_chunk (
    id          BIGSERIAL PRIMARY KEY,
    page_id     TEXT NOT NULL,
    page_type   TEXT NOT NULL,
    title       TEXT NOT NULL,
    url         TEXT,
    chunk_index INTEGER NOT NULL,
    chunk_text  TEXT NOT NULL,
    embedding   BYTEA,
    tsv         tsvector GENERATED ALWAYS AS (to_tsvector('english', chunk_text)) STORED,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX knowledge_chunk_tsv_idx ON knowledge_chunk USING GIN (tsv);
```

Base shape = spike's `knowledge_chunk` (id, page_id, page_type, title, url,
chunk_index, chunk_text, tsv+GIN) plus the `embedding` and `created_at` columns
this slice adds.

## Tests added

`tests/test_knowledge_config.py` (4 tests): default DSN
(`postgresql://toee:toee@localhost:5432/toee_knowledge`, never mentions
`toee_va`), env override, blank-env fallback, and a lazy-seam guard that
monkeypatches `psycopg.connect` to raise and proves `knowledge_database_url()`
never calls it.

`tests/test_knowledge_migrate.py` (9 tests):
- Static (always run, no DB): migrations ordered/non-empty starting at
  `0001_knowledge_chunk`; the migration SQL declares `CREATE TABLE
  knowledge_chunk`, a generated `tsvector`, a `USING GIN` index, and an
  `embedding` column; the knowledge migrations dir is a distinct directory
  from the business one (not nested either way); the business datastore's own
  migrations (`hermes_runtime.datastore.migrate.discover_migrations()`) never
  mention `knowledge_chunk` (S-ISO guard against contamination in the other
  direction).
- Live-DB (skip if unreachable): migration creates `knowledge_chunk` with all
  expected columns; idempotent on a second run; the GIN index actually exists
  via `pg_indexes` introspection; an inserted row is findable via
  `plainto_tsquery`; and — the isolation invariant — migrating the knowledge
  schema leaves the business schema (a separate fixture, separate DSN)
  without a `knowledge_chunk` table.

`tests/conftest.py`: factored the existing `temp_schema_conn` fixture's body
into a shared `_temp_schema(dsn, *, env_label)` generator and added
`temp_knowledge_schema_conn` on top of it, pointed at
`KNOWLEDGE_DATABASE_URL` instead of `DATABASE_URL`. `temp_schema_conn` itself
is otherwise unchanged (same behavior, same skip semantics).

## Self-review

- Isolation (S-ISO): no new tables in `hermes-runtime/migrations/` (business
  datastore); `hermes_runtime/knowledge/migrate.py` never imports or connects
  to `hermes_runtime.datastore`; verified live that `toee_va` has 0
  `knowledge_chunk` tables after running the knowledge migration.
- Lazy DSN: `knowledge_database_url()` is a pure `os.environ.get` read, no
  import-time or call-time psycopg touch — covered by a monkeypatch test that
  fails loudly if that ever changes.
- Followed the `datastore/config.py` + `migrate.py` pattern closely rather
  than inventing a different shape — no dev-seed-exclusion logic ported over
  since the knowledge store has no dev-seed concept (would have been
  unrequested complexity).
- No pgvector, as directed. `embedding BYTEA` is unused this slice
  (ingestion/retrieval are S07/S08) — documented as a deliberate deferral in
  the migration's SQL comment.

## Concern to flag

The 0.0.3 knowledge spike's `probe/ingest.py` already created an ad-hoc
`knowledge_chunk` table directly in this machine's `toee_knowledge` **public**
schema (167 real rows, columns `id, page_id, page_type, title, url,
chunk_index, chunk_text, tsv` — no `embedding`, no `created_at`, no
`schema_migrations` bookkeeping row). My tests never touch `public` (they run
in throwaway schemas via the fixtures), so this didn't block TDD or the full
suite. But running `python -m hermes_runtime.knowledge.migrate` for real
against this developer machine's `public` schema right now would fail with
"relation knowledge_chunk already exists" (no bookkeeping row marks 0001 as
applied). I did not paper over this with `CREATE TABLE IF NOT EXISTS` — that
would silently leave the table missing `embedding`/`created_at` rather than
failing loudly. The probe's own README calls it throwaway ("the DSN seam +
`knowledge_chunk` schema are the kept part"), so the fix is operational, not
code: either `DROP TABLE knowledge_chunk;` in `toee_knowledge` (then
`migrate()` applies cleanly and S07 re-ingests the real corpus) or manually
insert a `schema_migrations` row if the 167 rows must be kept as-is.
