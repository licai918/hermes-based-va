## 1. What was pinned, and how the surface was verified

### 1.1 The SDK call surface (closes `driver.py:585`)

The `ponytail:` said "verified at staging smoke; upgrade path is to pin the exact
Composio v3 method + envelope once confirmed against live toolkits." That is now
done, against **`composio` 0.15.0** (the hermes-runtime dependency):

| Pinned | Value |
|--------|-------|
| method | `Tools.execute(slug, arguments, *, connected_account_id=None, user_id=None, version=None, …)` — `slug`/`arguments` positional, the rest keyword-only |
| envelope | `ToolExecutionResponse` = a plain `dict` with exactly `{"data": dict, "error": str \| None, "successful": bool}` (the SDK `model_dump()`s the HTTP response before returning, so it is never a model) |
| version resolution | per call, from the SDK-level `toolkit_versions` map, keyed by the vendor toolkit slug |

The old code called `tools.execute(action, user_id=…, connected_account_id=…,
arguments=params)`. That happened to work, but nothing held it in place. Two
things hold it now:

- **`hermes-runtime/tests/test_composio_sdk_pin.py`** introspects the *installed*
  SDK — `inspect.signature(Tools.execute)`, `ToolExecutionResponse.__annotations__`,
  and the presence of `ToolVersionRequiredError()` in `Tools._execute_tool`. Zero
  network, runs in CI, fails the day an SDK bump moves the method or renames an
  argument. This is the whole point of "pinned": the driver stops guessing at
  staging-smoke time and starts failing in CI instead.
- **`python -m hermes_runtime.composio_smoke` phase 2** resolves every slug in
  `ACTION_MAPPING` against the **live** toolkit at its pinned version.

Two behaviour changes fell out of reading the envelope properly:

- `result.get("successful", False)` (was: only `is False`) — an envelope missing
  the key now fails closed instead of being treated as success.
- a non-`dict` `data` is a governed `composio_api_error` instead of being returned
  as the raw envelope. Previously that envelope reached a response mapper, which
  would `.get()` its way to an all-`None` "result" — a fabricated answer (ADR-0020).

### 1.2 Toolkit version pins

Pins are now **required**, not optional, and are keyed by **Composio's** toolkit
slug rather than ours:

```
COMPOSIO_TOOLKIT_VERSION_SHOPIFY    = 20260506_00
COMPOSIO_TOOLKIT_VERSION_QUICKBOOKS = 20260623_00   # NOT _QBO
COMPOSIO_TOOLKIT_VERSION_SQUARE     = 20260616_00
```

**The `QBO` → `quickbooks` mismatch was a live trap.** The SDK resolves a call's
version with `get_toolkit_version(tool.toolkit.slug, …)` and reads
`COMPOSIO_TOOLKIT_VERSION_<SLUG>` from the environment. Our toolkit key is `qbo`,
the vendor slug is `quickbooks` — so a `COMPOSIO_TOOLKIT_VERSION_QBO` would have
been silently ignored, the version would resolve to `"latest"`, and
`Tools._execute_tool` raises `ToolVersionRequiredError` **when `latest` is
resolved and `dangerously_skip_version_check` is not set**. That is a governed
failure, but it arrives mid-turn on a customer's message and names neither the
toolkit nor the variable. `TOOLKIT_SLUG` now carries the mapping explicitly, and
`pinned_toolkit_versions()` raises `configuration_missing` **at process boot**
naming the exact missing variable. `"latest"` is rejected as if unset.

Only toolkits that actually have a connected account need a pin, so a
Shopify-only deployment still boots.

The old `_toolkit_versions_from_env()` helper was deleted — the SDK's own
`get_toolkit_versions()` already reads those env vars, so the helper was
duplicating the SDK verbatim. The driver now passes the *validated* map
explicitly.

### 1.3 Per-call deadline (NFR-8)

There was no deadline on the Composio path at all. The SDK ships a **60 s** read
timeout plus `DEFAULT_MAX_RETRIES` — worst case ~180 s for one tool call, longer
than the SMS turn it is serving.

`COMPOSIO_DEADLINE_MS` (default **8000**) is now passed as the SDK `timeout`, with
`max_retries=0` so the deadline is a per-*call* bound, not a per-*attempt* one. On
expiry the SDK raises → the driver's existing `except Exception` converts it to a
governed `composio_api_error` → dispatch renders the Tool Unavailable Response.
Measured in the smoke: **~2.0 s to a governed failure** against a dead backend,
against an 8 s deadline (§4).

## 2. Deployment checklist — all five tool-executing processes

The slice said "the gateway AND both dispatch servers". Tracing it end to end,
the real surface is **five processes**, and the one the slice's framing omits is
the one that matters most:

| Process | Why it reaches Composio | compose service | Cloud Run |
|---------|------------------------|-----------------|-----------|
| **gateway** | Ingress Phone Match falls back to a live Shopify customer lookup (`hermes_runtime/datastore/shopify_identity.py`, `SHOPIFY_GET_CUSTOMERS_SEARCH` / `SHOPIFY_GET_ALL_CUSTOMERS`) | `gateway` | `toee-hermes-gateway` |
| **turn worker** | **Runs the customer's async agent turn — this is where `toee_shopify_read` / `toee_qbo_read` actually execute.** Since 0.0.4 S02 the gateway only enqueues | `turn-worker` | own service |
| **dispatch copilot (8091)** | The Copilot draft turn calls the same external reads | `dispatch-copilot` | own service |
| **dispatch admin (8092)** | Same deterministic `tools:dispatch` surface, `supervisor_admin` profile | `dispatch-admin` | own service |
| **background worker** | Shares the image and the `l6_review` fork's tool surface | `background-worker` | own service |

Locally all five inherit `hermes-runtime/.env` through the `x-hermes-runtime`
compose anchor, so one file covers them and `pnpm dev` needs nothing new. **On
Cloud Run each is its own service and each needs the variables set
independently** — a revision that flips `INTEGRATION_DRIVER=composio` on only
some of them is a split brain: the same customer gets live data on one path and
mock data on another.

### Required per service

```
INTEGRATION_DRIVER=composio
COMPOSIO_API_KEY                          (Secret Manager)
COMPOSIO_USER_ID
COMPOSIO_{SHOPIFY,QBO,SQUARE}_CONNECTED_ACCOUNT_ID
COMPOSIO_TOOLKIT_VERSION_SHOPIFY          required — boot fails without it
COMPOSIO_TOOLKIT_VERSION_QUICKBOOKS       required — note QUICKBOOKS, not QBO
COMPOSIO_TOOLKIT_VERSION_SQUARE           required
COMPOSIO_DEADLINE_MS                      optional, default 8000
TOOL_BACKEND=datastore                    pre-existing boot requirement
```

### Order of operations

1. Link/verify all three connected accounts; confirm **active** in the dashboard.
2. Set the variables above on **every** service in the table.
3. Run `python -m hermes_runtime.composio_smoke` from one of them; require PASS.
4. Roll the services. A `configuration_missing` at boot is a missing pin, not a
   vendor outage — the message names the variable.
5. **Rollback** = `INTEGRATION_DRIVER=mock` on every service in the table,
   together.

Checked-in form: `docs/ops/composio-connected-accounts.md` §6 (new), with the
`deploy-cloud-run.md` env table updated to match.

**Local cutover status:** the owner's `hermes-runtime/.env` already carries
`INTEGRATION_DRIVER=composio`, `TOOL_BACKEND=datastore`, all three connected
accounts, all three pins, and a live `COMPOSIO_API_KEY`. So the *local* stack is
cut over and was used to run the smoke below. **The Cloud Run revisions were not
touched — I have no deployment credentials** (§7).

## 3. Finding: Composio cannot create a Square payment link

This is what pinning was for, and it fired immediately.

Phase 2 of the smoke returned a hard 404 for the mapped slug:

```
[FAIL] toee_square_payment_link.send_payment_link -> SQUARE_CREATE_PAYMENT_LINK
       NotFoundError: 404 {'message': 'Tool SQUARE_CREATE_PAYMENT_LINK not found',
                           'code': 2401, 'slug': 'Tool_ToolNotFound'}
```

Follow-up probes against the live Composio API:

- the Square toolkit at pin `20260616_00` exposes **122 tools**; the only
  payment-link action is `SQUARE_RETRIEVE_PAYMENT_LINK` (read one) — there is no
  create, and no `SQUARE_CREATE_INVOICE` / publish-invoice path either.
- `SQUARE_CREATE_PAYMENT_LINK` also 404s at **`latest`**, so it is not a
  version-pin gap.
- a **catalog-wide** search for "payment link" returns
  `FLUTTERWAVE_CREATE_PAYMENT_LINK`, `RAZORPAY_CREATE_PAYMENT_LINK`,
  `STRIPE_CREATE_PAYMENT_LINK`, `POOF_CREATE_PAYMENT_LINK` — and, for Square,
  only `SQUARE_RETRIEVE_PAYMENT_LINK`.

The slug was never real. The original comment said slugs were "verified against
live **Shopify** toolkit (staging smoke)" — Shopify was verified; QBO and Square
were plausible guesses that nobody had called.

**Resolution:** `toee_square_payment_link.send_payment_link` now fails closed on
the Composio backend, using the same mechanism `get_ar_summary` already used —
generalised from two hand-written raiser functions into one `ActionSpec.unavailable`
field, checked in `ComposioDriver.execute` **before** the backend call and before
the connected-account check (so the message names the real reason). Net: two
functions deleted, one field added.

Customer-visible outcome is unchanged from the 404 path (governed Tool
Unavailable Response) — but the operator now gets a message that says why, and
the failure costs no round trip. Critically it is **not** a fallback to mock:
texting a customer a fabricated payment URL is the worst failure available here
(ADR-0020, FR-21).

`ponytail:` on the constant names the ceiling and two upgrade paths, laziest
first: a Composio **custom tool** wrapping `POST /v2/online-checkout/payment-links`,
or a **direct Square REST driver** as a per-tool overlay beside Composio — exactly
the shape FR-20 uses for EasyRoutes. Neither is S12 scope, and payment-link
creation is a financial write that wants its own slice.

## 6. Secret-scan gate (NFR-6, gap-review Q3)

`scripts/secret-scan.mjs` + one CI job (`secret-scan`, first in `ci.yml`).
**S14 references it; it does not re-implement it** — a new credential is one line
in `SECRET_VARS`, not a new job. `EASYROUTES_API_TOKEN` and `EASYROUTES_CLIENT_ID`
are already in the list, with a selfcheck case proving they are covered.

Scope is `git ls-files` — what is actually committed. Three rules:

1. **No tracked env file.** `.env` / `.env.local` are where real credentials live
   on every box in this repo; committing one is the likeliest leak and needs no
   pattern matching. Only `*.env.example` is allowed. (Belt to `.gitignore`'s
   braces — the point is that a `git add -f` cannot slip past CI.)
2. **Known public credential prefixes:** OpenRouter `sk-or-v1-`, Anthropic
   `sk-ant-`, OpenAI `sk-`+40, Shopify `shp{at,ca,pa,ss}_`+32hex, Slack `xox?-`,
   GitHub `gh[pousr]_`+36, AWS `AKIA`+16, Google `AIza`+35, and PEM private-key
   blocks. Documented prefixes only — a shape rule that guesses is one people
   learn to skip.
3. **A secret-named variable assigned a real value.** This is the rule that
   covers vendors with no recognisable prefix (Composio, SimpleTexting). Allowed
   values: empty, `<placeholder>`, `...`, `${VAR}` interpolation, `:-default`
   shell substitution, a code reference (`rt.INTERNAL_JOB_SECRET`), a
   SCREAMING_CASE constant, a Secret Manager ref (`composio-api-key:latest`), and
   — the load-bearing one — a **slug-shaped value**: only lowercase letters and
   dashes, ≤40 chars. `dev-webhook-token`, `copilot-tok`, `from-env` are human
   written; every provider's tokens carry digits, mixed case, or underscores.
   That single rule cleared 19 of 20 false positives across the repo's test
   fixtures and published dev defaults while `zz9realvaluepastedhere` still trips.

`node scripts/secret-scan.mjs --selfcheck` runs 20 positive/negative cases (a
gate nobody can prove still bites is a gate that quietly stops biting), and CI
runs it before the scan. Current repo: **clean**.

Known limits, stated rather than hidden: it scans the tree at the ref, not
history, and does not rotate anything. Failure text says so — "if a value is
real: ROTATE IT at the provider, then remove it from the tree".

## 9. What S13 and S14 must know

### S13 (`get_ar_summary`)

- **`_qbo_ar_summary_request` / `_qbo_ar_summary_response` no longer exist.** The
  fail-closed is now data: `ActionSpec(QBO, "QUICKBOOKS_GET_AGED_RECEIVABLES_REPORT",
  unavailable=_AR_SUMMARY_UNAVAILABLE)`. To implement the real path, drop
  `unavailable=` and supply `request_mapper=` / `response_mapper=`. The invariant
  test `test_action_mapping_covers_exactly_the_layer1_actions` enforces
  *exactly one* of {both mappers, `unavailable`} per entry, so a half-migration
  fails in CI rather than raising `TypeError` mid-turn.
- **`QUICKBOOKS_GET_AGED_RECEIVABLES_REPORT` was never probed** — the smoke skips
  gated-off slugs. Confirm it resolves at pin `20260623_00` before building on it;
  `QUICKBOOKS_QUERY_INVOICES` and `QUICKBOOKS_LIST_INVOICES` did resolve, so the
  QBO account and pin are good.
- The original fail-closed reasoning still stands and is the thing to solve: the
  Aged Receivables report is an **all-customer aggregate**, so a per-customer
  summary cannot be derived from it without misattributing balances. There is
  still no Shopify→QBO `CustomerRef` bridge; `_invoice_owned_by_verified` scopes
  by matching `shopify_customer_id` on the invoice and drops anything it cannot
  attribute.
- Add a happy-path entry to `SMOKE_CALLS` when it lands; the smoke test in
  `test_composio_sdk_pin.py` keeps it honest against `ACTION_MAPPING`.

### S14 (EasyRoutes driver)

- **The secret-scan gate exists. Do not build another.** Add credential names to
  `SECRET_VARS` in `scripts/secret-scan.mjs` — `EASYROUTES_API_TOKEN` and
  `EASYROUTES_CLIENT_ID` are already there, with a selfcheck case. The CI job is
  `secret-scan` in `.github/workflows/ci.yml`.
- **`INTEGRATION_DRIVER` is not the axis.** `rest` is gone. EasyRoutes is a

  bounds a call by passing an HTTP-client `timeout` plus `max_retries=0`, which

  and the `COMPOSIO_DEADLINE_MS` env shape are the template.
- **Deploy to all five processes** (§2), not just the gateway. Add EasyRoutes to
  runbook §6.2's variable table.
- PAC-6 wants an **EasyRoutes test delivery** for the same test customer created
  in §7 — coordinate with whatever the owner creates there rather than a second
  test entity.

### Anyone touching the Composio driver

- `hermes-runtime/tests/test_composio_sdk_pin.py` will fail on an SDK upgrade
  that moves `Tools.execute` or changes `ToolExecutionResponse`. That is the
  intent — read the diff, re-run the smoke, then update the pin.
- After **any** toolkit-version bump, re-run
  `python -m hermes_runtime.composio_smoke`. Phase 2 is cheap and is the only

## 10. Concerns

- **The smoke is FAIL and should stay FAIL** until the PAC-6 test customer
  exists. Nothing about the tooling should be "fixed" to make it green.
- **The Cloud Run cutover is unexecuted.** Local is cut over; production is not,
  and I could not verify a deployment I have no credentials for.
- **`toee_square_payment_link` is unavailable on the live backend** and there is
  no in-scope fix (§3). This is a real capability gap the roadmap should
  acknowledge — the product currently cannot send a Square payment link in
  production, and could not before this slice either; the difference is that now
  it says so.
- **The ingress phone-match scan is still unbounded in aggregate.**
  `shopify_identity._lookup_via_pagination` runs up to `_MAX_SCAN_PAGES = 40`
  sequential Composio calls **on the webhook ack path**. Each call is now bounded
  at 8 s, so the worst case is ~320 s — better than unbounded, still far past any
  ack budget. It only triggers when `SHOPIFY_GET_CUSTOMERS_SEARCH` is absent, and
  the pinned Shopify toolkit does expose it, so the scan should not fire in
  practice. Flagged, not fixed: capping the *aggregate* is its own change and
  belongs with whoever owns the ingress path.
- **`COMPOSIO_DEADLINE_MS=8000` is a guess**, chosen to sit well under a turn
