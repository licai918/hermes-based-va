# 0.0.4 S15 — `/admin/integrations` status page (FR-23)

Opens the T5 integrations-ops track: an **admin-only** page making each integration's
config/connection health visible *before* a customer's turn fails. Builds the page +
the governed admin read; renders "never probed"/"unknown" for the fields S16 (probes)
and S17 (reconnect) will fill.

## Uniform per-integration status shape

The governed read returns `{ active_driver, integrations: [row, ...] }`. Each row:

| field | meaning | S15 source |
|---|---|---|
| `key` | stable id (`shopify`/`qbo`/`square`/`easyroutes`/`simpletexting`/`openrouter`/`gadget`) | static |
| `label` | human name | static |
| `kind` | `composio_toolkit` \| `easyroutes` \| `simpletexting` \| `openrouter` \| `gadget` | static |
| `configured` | **green only if the credential is actually present** | see signal table below |
| `status` | `configured` \| `not_configured` (derived from `configured`) | derived |
| `pinned_version` | Composio toolkit version string, else `null` | `composio_config_status()` |
| `last_successful_call` | `null` everywhere → panel shows **"unknown"** | nothing records it yet |
| `last_probe` | `null` everywhere → panel shows **"never probed"** | S16 fills this |
| `detail` | human note (missing env-var **names** only, never values) | computed |

`active_driver` (`mock`/`composio`) rides alongside so an admin sees whether Composio
creds present actually route live yet.

### Which signal backs `configured` for each of the 7 rows

The read reuses each driver's **own existing** signal — no new health semantics invented:

- **Shopify / QBO / Square (Composio toolkits)** — new total, secret-free
  `composio_config_status()` (added beside `pinned_toolkit_versions`, S12's config
  surface): `configured` = `COMPOSIO_API_KEY` present **AND** that toolkit's
  `COMPOSIO_*_CONNECTED_ACCOUNT_ID` present **AND** a real (non-`latest`) version pin
  in `COMPOSIO_TOOLKIT_VERSION_<SLUG>`. `pinned_version` = that pin (a version, not a
  secret). Mirrors exactly what a live call needs, so green = would actually run.
- **EasyRoutes** — `drivers/easyroutes/driver.py::easyroutes_configured()`
  (`EASYROUTES_API_TOKEN` + `EASYROUTES_CLIENT_ID`).
- **SimpleTexting** — new `simpletexting_reply.py::simpletexting_configured()`
  (`SIMPLETEXTING_API_TOKEN`), sibling to `easyroutes_configured`.
- **OpenRouter** — new `openrouter.py::openrouter_configured()` (`OPENROUTER_API_KEY`).
- **Gadget mapping endpoint** — `drivers/gadget.py::gadget_configured()`
  (`GADGET_API_KEY`) — the paymentstatussync qboCustomerMapping bridge (S27), a T4
  integration too.

All signals read `os.environ` in the process serving the read. The admin dispatch
server (`dispatch-admin`, `supervisor_admin`) inherits the same docker-compose
`env_file` (`hermes-runtime/.env`) the tool-executing processes use, so the config
presence it reports is honest for the live deployment — owner-blocked (no key) shows
`not_configured`, never a fabricated `healthy`.

**Integrations shown:** Shopify, QBO, Square (Composio toolkits), EasyRoutes,
SimpleTexting, OpenRouter (the PAC-7 six) **plus** the Gadget mapping endpoint = 7 rows.

## Tool registration checklist (0.0.3 S31 lesson — all 4 steps)

New governed tool `toee_integrations` with one action `get_integrations_status`,
registered end-to-end (template: S05's `toee_job_queue`):

   `toee_integrations__get_integrations_status`; `schemas.py` needs no `PARAM_SCHEMAS`
   entry (no params). `test_plugin_manifest.py` proves manifest == catalog.

   `toee_job_queue`), plus the test source-of-truth `ADR_0038_SUPERVISOR`.

   `plugin/__init__.py::_AGENT_EXCLUDED_ACTIONS` (agent-excluded, **not** in
   `_TOOL_PARAM_CONVENTIONS`) — exactly the `toee_job_queue`/`toee_retention` pattern.

Datastore handler: `hermes-runtime/.../datastore/handlers/integrations.py`, merged into
`build_datastore_registry()`. A deterministic mock stub (`drivers/mock/admin_stubs.py`)
keeps the catalog-coverage tests green and gives pure-mock dev an honest (all

## Admin-only role rationale (ADR-0093, gap-review P4 — deliberate asymmetry)

`/admin/*` is supervisor+admin (an operations tier). The integrations page + its BFF
are narrowed to **admin-only** via a new `requiresAdmin`/`isAdminOnlyPath` branch in
`lib/auth/access.ts` (one place gates both page nav and the API route through
`canAccess`). Rationale, stated in-code: **integrations are a credential surface**, so
they are deliberately narrower than the dead-letter **operations** view (supervisor+
admin) — a supervisor triaging stuck work should not need to see every backend's
credential-configuration status. Nav: an admin-only "Integrations" item appended for
the admin role only (a supervisor must not see a 403 link).

## Surface / API-only (S09)

Workbench stays API-only: the BFF (`lib/bff/admin/integrations.ts`) dispatches over the
Supervisor Admin Profile API via `hermes-api-client` (`createAdminApiClient`), mapping
the governed JSON with a strict mapper (a missing `configured` **throws** rather than
defaulting true — an owner-blocked integration can never be painted green by a shape
drift). Missing API config still fails closed at boot (unchanged). Page shell mirrors
the dead-letter sibling.

## What shows "never probed" / "unknown" pending S16

- **Last probe result** → `last_probe: null` → panel renders **"never probed"**. The
  handler's `conn` is accepted and unused today; S16 adds the newest-probe-row SELECT
  in this one handler (the deliberate seam).
- **Last successful call** → `last_successful_call: null` → panel renders **"unknown"**.
  Nothing in the drivers records a last-success timestamp yet — not fabricated.

## How secrets are kept out (NFR-6)

Only booleans, the Composio version-pin **string** (a version, not a credential), and
env-var **names** (in `detail`) ever leave the handler — never a token/key/account-id
value. `composio_config_status()` is total and returns no secret. A dedicated unit test
stuffs a sentinel secret into every credential env and asserts it never appears in the
serialized output. `node scripts/secret-scan.mjs` (+ selfcheck) stays clean.

## What S16 (probes) and S17 (reconnect) need from this shape

- **S16** fills `last_probe` per row from a scheduled probe job on the T2 recurring
  queue (FR-24). It reuses the same `*_configured()` signals (cheap gate before a live
  call) and the drivers' health affordances (`gadget.health()`, and a per-toolkit
  Composio/EasyRoutes/SimpleTexting/OpenRouter authenticated read). The SELECT lands in
  `handlers/integrations.py`; the row/BFF/panel already carry `last_probe` (+ status
  badge) so no shape change is needed — just populate it and (optionally) widen `status`
  beyond `configured`/`not_configured` to reflect a failed probe badge.
- **S17** keys reconnect off the stable per-integration `key` + `kind` this shape
  already exposes (Composio OAuth re-auth vs EasyRoutes token replacement), and the
  integrations ADR (deferred to S17) records the callback design.
