## The command

```bash
pnpm dev
```

`scripts/dev-up.mjs`, wired into the root `package.json` as `dev` (and
`dev:stack` for `--stack-only`). Node rather than a `.ps1` plus a parallel
`.sh`: Node is already a hard prerequisite (pnpm), and one file works in
PowerShell, Git Bash and Linux CI without a second implementation to keep in
sync.

### What it starts

Four phases. Everything except the workbench runs in `docker compose`; the
workbench dev server runs on the host, where its `node_modules` and hot reload
already live.

| Phase | Service | Where | Port |
| --- | --- | --- | --- |
| 1 | Postgres — `toee_va` **and** `toee_knowledge` | container `toee-va-postgres` | 5432 |
| 2 | migrations for both databases + dev seed | one-shot container `toee-va-migrate` | — |
| 3 | `internal_copilot` dispatch server | `toee-va-dispatch-copilot` | 8091 |
| 3 | `supervisor_admin` dispatch server | `toee-va-dispatch-admin` | 8092 |
| 3 | gateway (inbound SMS) | `toee-va-gateway` | 8080 |
| 3 | `turn_worker` | `toee-va-turn-worker` | — |
| 3 | `background_worker` | `toee-va-background-worker` | — |
| 4 | workbench dev server | host | 3000 |

New compose services: `migrate`, `dispatch-copilot`, `dispatch-admin`,
`gateway`. The two worker services from S02/S04 were refactored onto the same
YAML anchors (`x-hermes-runtime`, `x-datastore-env`) so all six share **one**
image, `toee-hermes-runtime:local` — the `uv sync --frozen` is paid once (~4 min
cold, cached afterwards) instead of six times.

The refactor preserved every existing key, including the
`stop_grace_period: 310s` on both workers and its comment tying it to
`job_queue.DEFAULT_LEASE_SECONDS` (300 s). It also **fixed** a latent bug: the
turn worker had no in-network `KNOWLEDGE_DATABASE_URL`, so knowledge retrieval
inside that container resolved the localhost default, which is not the compose
Postgres. Both workers now get it from the shared anchor.

The compose project name is untouched (derived from the directory), so the
existing `hermesbasedva_toee_pgdata` volume — the owner's data — is the same one
this uses.

## How readiness is determined

The brief's point: a `docker compose up` that returns before a service answers
produces exactly the confusing failures this slice removes. Each phase blocks on
a real signal.

| Service | Readiness signal |
| --- | --- |
| Postgres | `docker compose up --wait` against the existing `pg_isready -U toee -d toee_va` healthcheck |
| Migrations | one-shot container must exit 0 (`docker compose run --rm --build migrate`); output is streamed, so you see which versions applied |
| dispatch-copilot / dispatch-admin / gateway | compose healthcheck polls `GET /healthz` **inside** the container (`python -c urllib.request…` — no curl in `python:3.11-slim`), then the script *also* polls the published port from the host, so a container that is healthy but unreachable from Windows still fails loudly. Gateway gets `start_period: 30s` because `warm_knowledge_embedder()` runs at boot. |
| turn-worker / background-worker | no HTTP surface, so the signal is container state: after `up --wait` the script asserts `docker compose ps` reports `running` for both. A worker that fails closed on a missing `TOOL_BACKEND` lands in `restarting`, and the script dies naming the log command. |
| workbench | polls `GET http://127.0.0.1:3000/login` (180 s budget) before printing the login banner |

Phases 1 and 2 are separate `docker compose` invocations on purpose:
`up --wait` and a one-shot service that exits do not mix well, and running
`migrate` explicitly means its output is visible rather than buried in a
dependency graph.

## Environment: what it sets vs. what it requires

Two gitignored files, one direction of flow, no third place to look.

**`apps/workbench/.env.local` — source of truth, written by the script if absent.**
The workbench reads it directly (S09 fail-closed boot), and the script reads the
same values back out and exports them into compose's interpolation environment,
so the BFF's bearer and the server's `DISPATCH_API_TOKEN` cannot drift:

| Variable | Consumed by | Also becomes |
| --- | --- | --- |
| `HERMES_COPILOT_API_URL/TOKEN` | workbench BFF | `dispatch-copilot`'s `DISPATCH_API_TOKEN` |
| `HERMES_ADMIN_API_URL/TOKEN` | workbench BFF | `dispatch-admin`'s `DISPATCH_API_TOKEN` |
| `TEXTLINE_WEBHOOK_SECRET` | simulator route (signs) | gateway's `TEXTLINE_WEBHOOK_SECRET` (verifies) |
| `SIMULATOR_GATEWAY_URL`, `WORKBENCH_SESSION_SECRET` | workbench only | — |

The script also warns (does not fail) if `HERMES_*_API_URL` does not point at
:8091/:8092, and dies with a pointer if any of the four required pairs is
missing.

**`hermes-runtime/.env` — real credentials, mounted into every Python container
via `env_file`.** Required for a *fully* working stack, not for boot:

| Variable | If missing |
| --- | --- |
| `OPENROUTER_API_KEY` | copilot chat / inbound turns have no live model (S09 deleted the TS stub) |
| `INTERNAL_JOB_SECRET` | script substitutes `dev-internal-job-secret` |
| `REPLY_SENDER` | script substitutes `simulated` |
| `INGEST_CORPUS_PATH` | a queued `ingest` job dead-letters immediately, naming the variable (unchanged, still commented in compose) |
| `AGENT_EXPERIENCE_LEARNING` + key | an `l6_review` job fails loudly by design (unchanged, still commented in compose) |

**Set by compose, not by a human:** `TOOL_BACKEND=datastore` on the gateway and
both workers (the S02 boot requirement — they fail closed without it) and on both
dispatch servers; the two in-network DSNs; `HERMES_APPLY_DEV_SEED=1` on the
migrate step only; `REPLY_SENDER=simulated` as the dev default, which also
unlocks the simulator's simulated-only `link_identity` action
(`tool_dispatch_composition._SIMULATED_ONLY_ACTIONS`).

Precedence is deliberate: compose `environment:` beats `env_file:`, so the script
resolves each shared secret from `.env`/`.env.local` **first** and only falls back
to a dev default. Nobody's real `INTERNAL_JOB_SECRET` gets clobbered by running
`pnpm dev`.

### Two preflights that remove known local failures

- **Docker unreachable.** The owner's box has a persistent user-level
  `DOCKER_HOST=tcp://localhost:2375` that Docker Desktop does not serve; every
  `docker` call dies with `dial tcp [::1]:2375`. The script probes
  `docker version`, and if it fails *and* `DOCKER_HOST` is set, drops the
  variable for its own child processes and retries. If it still fails it says
  "start Docker Desktop" instead of surfacing the raw connect error.
- **Port already allocated.** Any compose failure prints a hint that a
  hand-started server from an earlier session probably owns the port, with the
  command to find and kill it.

## Stale hand-started servers

Three things, so the owner cannot walk into the `Unknown tool "toee_job_queue"`
trap and read it as a bug:

1. **Different ports.** The composed servers use **8091/8092**, which is what the
   owner's `apps/workbench/.env.local` already points at. A leftover on the old
   8081/8082 therefore cannot silently answer for a composed one — it just sits
   there. A leftover on 8091/8092 *does* collide, and compose fails with
   `port is already allocated`, which the script annotates with the fix.
2. **A named section** in the workbench README —
   "Stale hand-started servers" — with the exact symptom, `Get-NetTCPConnection`
   + `Stop-Process` for PowerShell and `npx kill-port` for Git Bash. Every other
   runbook links to it.
3. **Proof the composed one is current**, in `local-dispatch-servers.md`:

```
POST http://127.0.0.1:8092/v1/tools:dispatch  {"tool":"toee_job_queue","action":"list_dead_letters","params":{}}
-> {"ok":true,"data":{"jobs":[],"outbound":[],"recent_replays":[]}}
```

Verified live (below). The pre-S05 server the owner had on :8092 answers the same
call with `Unknown tool "toee_job_queue"`.

At the time of this run nothing was listening on 3000/5432/8080/8081/8082/8091/8092
and Docker Desktop was stopped, so there was nothing stale to kill; the guidance is
written for the next time.

## What S25 can reuse for CI

- **`pnpm dev:stack`** is the whole thing: it is `pnpm dev` minus the workbench
  dev server, and it exits 0 once every service is *ready*, not merely started.
  A CI job can run it and then run tests against :8091/:8092/:8080. That covers
  NFR-9's "gateway + both dispatch servers + turn worker + Postgres" verbatim.
- The script is plain Node with no dependencies and no PowerShell, so it runs on
  `ubuntu-latest` unchanged. `dockerPreflight()`'s `DOCKER_HOST` handling is a
  no-op where the variable is unset.
- The `migrate` compose service is the migration step: idempotent, applies both
  databases, self-provisions `toee_knowledge` (CI Postgres only provisions
  `toee_va`). `HERMES_APPLY_DEV_SEED=1` is set on that service; drop it for a
  seed-free CI database by overriding the variable.
- Healthchecks live in `docker-compose.yml`, so `docker compose up --wait` gives
  CI the same readiness gate without reimplementing the polling.
- If CI needs different ports or tokens, the compose file takes them by
  interpolation (`${HERMES_COPILOT_API_TOKEN:-dev-copilot-token}` etc.) — no file
  edit needed.
- Nothing in the CI workflow was touched (S25 owns the topology).

## Fix wave 1

Seven review findings against this slice (2 Important, 5 Minor) — all traced
back to the slice's own purpose: eliminating confusing half-started stacks.
Findings 1 and 2 were both ways a stack could still look "up" while quietly
broken. Fixed on `feat/0.0.4-land-all` as follow-up commits.

### Finding 1 [Important] — a crash-looping worker passed as ready

`assertWorkersRunning()` sampled `docker compose ps` once and accepted
`"running"` — true for a worker crash-looping on a bad DSN or missing
credential most of the time (`restart: unless-stopped`, no healthcheck), and
also true for a worker still inside `resolve_turn_collaborators()` (Postgres
connect + embedder warm) and not yet consuming jobs.

Replaced with `waitForWorkersReady()`, which polls (60s budget) and requires
all three of:

1. `docker compose ps` succeeded and reports `running` (not `restarting`/absent).
2. `docker inspect -f '{{.RestartCount}}' toee-va-<service>` is `0` — a
   crash-loop shows up here even between two "running" samples, which a single
   `ps` snapshot can miss.
3. `docker compose logs <service>` contains `polling for` — both
   `turn_worker.main()` and `background_worker.main()` log
   `"... polling for ... jobs every ...s"` the instant they finish collaborator
   setup and enter the poll loop (grepped the actual log lines in
   `hermes_runtime/turn_worker.py:285-290` and
   `hermes_runtime/background_worker.py:375-381` — wording differs
   ("Turn worker" vs "Background worker") but both match `/polling for/`).

Failure messages name the specific worker and point at
`docker compose logs <service>`, and distinguish "not running" from
"running but crash-looping" from "running but not yet polling."

Parsing logic (`parseComposeStates`, `hasStartedPolling`) is pulled out as pure
functions and covered by `--selfcheck`. The end-to-end polling loop itself
needs live containers; verified live (below).

### Finding 2 [Important] — env precedence made real inbound webhooks silently 401

`resolveDevEnv()` resolved `TEXTLINE_WEBHOOK_SECRET` as
`wb.TEXTLINE_WEBHOOK_SECRET || rt.TEXTLINE_WEBHOOK_SECRET || "dev-webhook-secret"`
— workbench `.env.local` first. Since `.env.local` is auto-written with
`TEXTLINE_WEBHOOK_SECRET=dev-webhook-secret` the first time `pnpm dev` runs and
essentially never absent afterward, an operator who followed
`docs/ops/local-gateway.md:249` ("set `TEXTLINE_WEBHOOK_SECRET` in
`hermes-runtime/.env`") would see it silently ignored: the gateway received
`dev-webhook-secret` regardless, and every real (ngrok) inbound 401'd with no
signal anything was wrong.

**Decision: `hermes-runtime/.env` wins.** Reasoning: `hermes-runtime/.env` is
the real-credentials file (`OPENROUTER_API_KEY`, `TEXTLINE_ACCESS_TOKEN`,
`INTERNAL_JOB_SECRET` all live there and are gitignored/never-committed);
`apps/workbench/.env.local` is a local-only file `dev-up.mjs` auto-generates
with dev defaults. A secret that must match an *external* provider belongs to
the "real credentials" file by nature — same reasoning that already put
`INTERNAL_JOB_SECRET` and `REPLY_SENDER` on `rt`-first precedence, which this
change now matches (removing the inconsistency finding 2 also named). For the
common case (no ngrok, simulator only, nobody touches
`hermes-runtime/.env`), behavior is unchanged: both files still fall back to
the same `"dev-webhook-secret"` default, so `wb`'s and the gateway's values
still agree.

Extracted the precedence into `resolveTextlineSecret(rt, wb)`, returning
`{ value, source }`. `dev-up.mjs` now logs which file won on every run:

```
[dev-up] TEXTLINE_WEBHOOK_SECRET resolved from apps/workbench/.env.local -- set it in
         hermes-runtime/.env to the real provider's secret before pointing ngrok/Textline
         at this gateway, or every inbound is a silent 401.
```

(silent no more — this is the "make a mismatch visible" requirement.) Updated
`docs/ops/local-gateway.md` (env table + the `TOOL_BACKEND=datastore` line) and
`apps/workbench/README.md` (`.env.local` table row) to state the actual
precedence.

Verified: `--selfcheck` asserts `resolveTextlineSecret` for all three cases
(rt set / wb set / neither). Verified live: ran `pnpm dev --stack-only` with no
`TEXTLINE_WEBHOOK_SECRET` in `hermes-runtime/.env` and observed the log line
above naming `apps/workbench/.env.local` as the source, confirming the wiring
end to end (did not temporarily edit the owner's `hermes-runtime/.env` to
avoid any risk of a stray change lingering there).

### Finding 3 [Minor] — the escape hatch bound a retired port

`run-dispatch-server.ps1`'s `-Port` defaulted to `8081` (the port S10 retired).
There is no single correct default across the script's three profiles anyway
(`internal_copilot`→8091, `supervisor_admin`→8092,
`customer_service_external`→ no composed port at all), so a different
hardcoded default would just be a different wrong guess for two of the three.
Made `-Port` `Mandatory` instead — matches how both header examples already
call it (always passing `-Port` explicitly) — so running the script with
defaults now fails loudly asking for a port rather than silently binding one
nothing talks to.

### Finding 4 [Minor] — a failed probe misdirected the reader

`assertWorkersRunning()` never checked `spawnSync`'s `status`, so any
`docker compose ps` failure (e.g. Docker Desktop hiccup) surfaced as
`turn-worker is "absent", not running` — blaming the worker for a probe
failure. `waitForWorkersReady()` now checks `ps.status !== 0` first and dies
with a message that names the probe itself
(`` `docker compose ps` failed: ... `` / "not a signal about
turn-worker/background-worker") before it ever looks at worker state.
Covered indirectly by the same live run as finding 1 (the probe succeeded
every time, as expected); the branch itself is a one-line status check with
no interesting logic to unit-test.

### Finding 5 [Minor] — build context included ~460 MB of agent scratch

`.dockerignore` excluded `.venv`/`node_modules`/`.git` but not `.claude/`
(~278 MB), `.tools/` (~32 MB), or **`.codegraph/`** (~153 MB — same category,
not named in the finding but found while checking "any other large scratch
directory in the same situation"; it's already gitignored as
"generated, per-machine"). Also added `.scratch/` and `.cursor/` (small, but
same kind of agent/editor scratch, already gitignored). Since the build
context is the repo root and phase 2 runs `--build` on every `pnpm dev`, this
was a build-context tax paid daily. Verified live: re-ran `pnpm dev --stack-only`
after the change and `docker compose run --rm --build migrate`'s own buildx
output reported `[internal] load build context / transferring context: 39.19kB`
— down from what would have been several hundred MB.

### Finding 7 [Minor] — a dead workbench polled for three minutes

`child.on("exit", ...)` for the workbench dev server was registered AFTER the
180s `waitForHttp` call, so a workbench that died at boot left the script
polling a dead port for the full timeout before reporting anything. Moved the
listener to immediately after `spawn()`, guarded by a `workbenchReady` flag: if
the child exits before the flag is set, the script dies immediately naming the
exit code instead of waiting out the timeout.

Verified live with a standalone repro of the exact pattern (spawn a child that
