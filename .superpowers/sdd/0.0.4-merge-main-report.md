# 0.0.4 — merge `origin/main` into `feat/0.0.4-land-all`

**Merge:** `git merge origin/main` (true merge commit; 68 ours + 22 theirs over
merge-base `b4abdcd`).
**Shape of the collision:** main retired Textline (provider rename + fail-closed
`REPLY_SENDER`), 0.0.4 replaced the transport (durable Postgres queue, two worker
processes, API-only workbench). The two are orthogonal in intent and heavily
overlapping in files — 26 conflicted paths.

Resolution rule applied throughout: **main's provider retirement is settled and
authoritative; our substrate work is the feature being integrated.** Where an
incompatibility was real (not just naming), the side whose *decision* the other
had not seen wins, and the loser's test is rewritten to assert the surviving
behaviour rather than deleted.

### 1b. The three that needed real judgment

#### (i) `hermes-runtime/hermes_runtime/turn_runner.py` — the one that could have gone wrong silently

Both sides rewrote the same eight lines for different reasons:

* **ours (S03/FR-12):** wrap the *entire* delivery step — `reply_sender` **and**
  the `on_reply_sent` mirror — in one `deliver_once` closure, with
  `OutboundMirrorFailed` distinguishing "customer has the reply, the Workbench
  row is missing" from "the send failed".
* **main (`a42a345`, ADR-0153):** **do not call `reply_sender` at all on an email
  turn.** The SimpleTexting client strips its first argument to digits, so an
  email `conversation_id` that happens to look like a phone number becomes a
  real, billable SMS to a caller-chosen number.

A naive "ours wins" here would have re-introduced main's bug inside our wrap.
**Both preserved:** the channel test moved *inside* `deliver()`
(`turn_runner.py:151-176`), so on email the mirror is the whole delivery and it
still sits in the one wrap. `TEXTLINE_SMS` → `SIMPLETEXTING_SMS`; the docstring's
"the real Textline POST" → "the real SimpleTexting POST".

Two tests encoded the pre-fix behaviour and were changed deliberately, preserving
**main's** intent (no SMS POST on an email turn):

* `tests/test_outbound_send.py::test_a_replayed_email_turn_writes_no_second_mirror_row`
  — `assert sent == [("conv-email-s03", …)]` → `assert sent == []`, with a comment
  naming ADR-0153/RK-4 as the reason.
* `tests/test_gateway_reply.py::test_email_turn_mirrors_its_reply_without_a_provider_send`
  (main's test) — its `SimpleNamespace` context gained `event_id="evt-email-1"`.
  Our wrap needs the event identity for the idempotency key; adding it preserves
  **our** intent without weakening main's assertion (`assert sent == []` still
  runs, and is now the thing the wrap must not break).

#### (ii) `hermes-runtime/hermes_runtime/gateway_composition.py` — two fail-closed guards, one of them now unreachable

Main added `_require_in_memory_store_is_allowed()` (ADR-0153: refuse the
per-process dedup store when `DEPLOY_ENVIRONMENT` is a deployed value, because an
unsigned webhook's only replay protection is `messageId` idempotency). S02 added
`_require_datastore_backend()`, which is **strictly stronger** — the gateway and
the worker are two processes, so the in-memory store can never work anywhere.

Deleting main's guard was tempting and wrong: it carries an ADR-0153 mitigation
row and a `deploy-cloud-run.md` runbook line. **Kept both**, with the redundancy
documented in place (`gateway_composition.py:152-172`) — `_require_datastore_backend`
sits upstream on all three entrypoints, and `_require_in_memory_store_is_allowed`
now guards the one remaining seam that can reach the in-memory branch without an
entrypoint: a direct `resolve_turn_collaborators()` call.

Two of main's tests had to move as a result:

* `test_build_gateway_app_refuses_in_memory_dedup_in_a_deployed_environment` —
  still passes (our error message contains `TOOL_BACKEND=datastore`), but the
  comment now names *which* guard fires, so the test does not lie about why.
* `test_build_gateway_app_allows_the_in_memory_store_for_local_development` —
  **directly contradicts invariant 5.** Replaced with
  `test_resolve_turn_collaborators_refuses_in_memory_dedup_when_deployed`, which
  unit-tests main's guard on the seam it still protects (deployed → raise, unset →
  allow). Our intent wins because the substrate cutover is what this merge is
  integrating; main's guard keeps its coverage.

Also merged in from main: `install_access_log_redaction()` at the top of
`build_gateway_app` (`gateway_composition.py:281-285`). The webhook token rides in
the URL because SimpleTexting offers no header option, so uvicorn's access log
would otherwise write the credential verbatim into Cloud Logging. Our S02 rewrite
of that function would have dropped it.

#### (iii) `hermes-runtime/hermes_runtime/gateway_app.py` — two idempotency mechanisms that look like one

Main added `store.claim_event(decision.event.event_id)` to the opt-out branch;
ours added the `deliver_once(slot=OPT_OUT_SLOT)` wrap around the confirmation.
They read as duplicates and are not:

* `claim_event` is a durable **compare-and-set on the inbound event** — it stops a
  verbatim webhook replay before it reaches the sender (unsigned webhooks, ADR-0153);
* `deliver_once` is the durable **record of the send itself** — it makes the
  outbound at-most-once even when the process dies mid-POST, and it is where a
  burned key is recorded for the S05 dead-letter view.

**Both kept**, with the reasoning written next to them (`gateway_app.py:205-231`).
Other hunks: `logger` kept, TGP signature headers dropped for main's
`WEBHOOK_TOKEN_QUERY_PARAM`, `parse_textline_fields` → `parse_simpletexting_fields`,
and our fix-wave-2 blank-`event_id` guard retained (it computes `fields` once and
passes it to `process_inbound`). Now-dead `import os` / `import time` removed —
they existed only for the TGP staleness window main deleted.

## 6. Concerns

* **`scripts/dev-up.mjs` rewrites nobody's existing `apps/workbench/.env.local`.**
  A developer with a file already on disk keeps `TEXTLINE_WEBHOOK_SECRET` in it,
  which the new resolver ignores — so `pnpm dev` falls through to the
  `dev-webhook-token` default and every simulated inbound silently 401s until they
  delete the file. The selfcheck cannot see this (it tests the resolver, not the
  developer's disk). A one-line "found TEXTLINE_WEBHOOK_SECRET, that variable is
  retired" warning in `resolveDevEnv` would close it; not added, because it is
  new behaviour rather than merge reconciliation.
* **`_require_in_memory_store_is_allowed` is defence-in-depth only.** No current
  entrypoint can reach it — `_require_datastore_backend` fires first on all three.
  Kept for its ADR-0153/runbook trail and unit-tested directly. If a future slice
  proves it is genuinely unreachable, delete it *and* the ADR-0153 mitigation row
  together.
* **`docs/adr/0105-cloud-tasks-async-textline-agent-jobs.md` keeps its Textline
  filename**, matching main's treatment of every other historical Textline ADR
  (main renamed code, not ADR records). The banner documents both supersessions.
* **The employee-confirmed governed send is outside the `deliver_once` wrap** — see
  the invariant-2 note in §2. Unchanged from both parents, flagged so it is a
  decision rather than an oversight.
