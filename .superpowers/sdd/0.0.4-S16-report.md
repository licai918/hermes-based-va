# 0.0.4 S16 — Scheduled integration health probes (FR-24)

**Branch:** `feat/0.0.4-land-all`
**Delivers:** FR-24. Closes ADR-0136's "lazy discovery only" gap: an expired/missing
credential is now caught by a background cadence, not by a failing customer turn.
**Status:** DONE. The scheduling, storage, the not-configured path, and the
fail-closed error path are all built and tested against fake probes + live Postgres.
The seven live wire calls are owner-blocked (no keys) and isolated to the same
UNVERIFIED path the T4 drivers already carry — a wrong guess records `failed`, never
a false `healthy`.

---

## Schedule wiring (the S01/S04 mechanism, reused — no new cron)

`integration_probe` is a NEW recurring type on the SAME `(type, window)` dedupe-key
tick S04 already runs, mirroring exactly how `retention` is scheduled:

| | Value |
|---|---|
| **Type** | `INTEGRATION_PROBE_JOB_TYPE = "integration_probe"` (next to the other type constants in `job_queue.py`; still a plain string, the queue imports nothing type-specific) |
| **Interval** | **900 s (15 min)** — `INTEGRATION_PROBE_INTERVAL_SECONDS` in `background_worker.py`, added to `SCHEDULES` beside retention |
| **Window** | `floor(epoch / 900)` → dedupe key `schedule:integration_probe:<window>` |
| **Worker allowlist** | `BACKGROUND_JOB_TYPES` gains `INTEGRATION_PROBE_JOB_TYPE` (now 4). The turn worker's allowlist is unchanged, so it still can NEVER claim it (FR-9 isolation preserved). |
| **Body** | `job_bodies()` maps `integration_probe → run_integration_probe_job` (lazy import, like the other bodies) |

**Why 15 min, and the cost.** Retention runs at 24 h because its windows are
730/90 **days**; a health probe wants to catch an expired credential *quickly*, so
minutes not a day. 15 min bounds the `/admin/integrations` badge's staleness to a
quarter-hour while writing ~96 `job` rows/day (vs retention's 1). The cost of a
shorter interval is linear `job`-row growth on a table whose retention is still
blocked on replacing the dedupe guarantee (ADR-0153 fix #5) — named as a `ponytail:`
ceiling at the constant, with the upgrade path (shorten toward 5 min if faster
detection is worth the growth).

**Dedupe preserved.** The window (900 s) is far larger than the worker's
`POLL_SECONDS` (5 s), so every tick inside one 15-min window derives the same dedupe
key → the unique index makes a duplicate tick (or a worker restart mid-window) a
no-op: exactly one probe job per window, no double-probe. Pinned by
`test_the_shipped_schedules_are_daily_retention_and_15min_integration_probe` and the
existing `test_ticking_twice_in_the_same_window_creates_exactly_one_job` mechanism.

`integration_probe` is NOT added to `REPLAY_BLOCKED_JOB_TYPES` /
`NON_CONCURRENT_JOB_TYPES`: re-probing is idempotent and concurrency-safe (two
cycles just write two sets of rows; the page reads the latest), so it inherits the
default "safe to replay" correctly.

## What each of the seven probes calls (the cheap authenticated read)

One AUTHENTICATED read per integration, each raising on any fault. All are gated by
the driver's own `*_configured()` env-presence check first (the same signal the S15
page uses), so a live call is only attempted when the credential is present.

| key | check | what it calls |
|---|---|---|
| shopify / qbo / square | `probe_composio_toolkit(<key>)` (new, in `composio/driver.py`) | Composio SDK `connected_accounts.get(<id>)` — a connected-account status read, no action execution, no vendor cost, no customer data |
| easyroutes | `EasyroutesDriver.health()` (new) | bounded `fetch_deliveries(order_number="__integration_probe__")` — a sentinel that matches no real delivery, so it never pulls a customer's data; empty list = reachable+authorized |
| simpletexting | `probe_simpletexting_token()` (new) | authenticated `GET {base}/api/account` — validates the token, never a send |
| openrouter | `probe_openrouter()` (new) | authenticated `GET {base}/key` — validates the key at zero token cost |
| gadget | `build_qbo_attribution().health()` (already existed) | bounded sentinel `qboCustomerMappings` query; empty = reachable+authorized |

**OpenRouter `/key`, not `/models` (a deliberate, documented deviation from the P1
note).** The gap-review note suggested the models endpoint. But `GET /models` does
NOT require auth on OpenRouter — it returns the public catalogue for a missing or
expired key alike, so it cannot tell a live key from a dead one and would paint a
rotated key "healthy". A false healthy is exactly the lie this track's spine forbids.
`GET /key` returns *this key's* own metadata, requires the key, and costs no tokens —
so it actually detects an expired/revoked key. Documented at the call site.

Every live wire detail (the Composio SDK connected-account surface, the SimpleTexting
account path, the OpenRouter/EasyRoutes/Gadget endpoints) is **UNVERIFIED** and
isolated exactly like `_HttpGadgetClient`/`_ComposioSdkClient` — confirmed at cutover
with the owner's keys. A wrong guess raises → recorded `failed`, never a false `ok`.

## The three honest states — not_configured ≠ failed ≠ healthy

Recorded in `integration_probe.py::_run_one`, one row per integration per cycle:

- **`not_configured`** — `configured()` is False, so the probe is **SKIPPED**. This
  is the owner-blocked reality for all seven today. It is NOT `failed` (the owner
  simply hasn't supplied the key) and NOT `ok`. Reason names the env var to set.
- **`failed`** — the credential IS present but the read raised: a governed
  `ToolDriverError` (401/timeout/vendor), a deadline timeout, OR any other exception.
  Per the empty-vs-error lesson, each driver's check treats an ambiguous/empty
  response as a raise, so an ambiguous read lands `failed`, never a false `ok`.
  `reason` carries a short, secret-free string.
- **`ok`** — the authenticated read returned cleanly.

These map straight to the page badge (see below) and to the log: a `failed` probe
emits a structured **ERROR** line (`integration probe FAILED key=… reason=…`,
alert-greppable); `not_configured` logs at INFO (expected owner-blocked state, not an
alert); `ok` at INFO. No paging/email in v1 (out of scope).

Distinctly pinned:
`test_not_configured_is_skipped_never_failed_never_ok`,
`test_a_clean_check_is_ok`, `test_a_governed_error_is_failed_with_its_reason`,
`test_an_arbitrary_error_is_failed_never_ok`,
`test_the_seven_real_probes_are_all_not_configured_when_owner_blocked`.

## Deadline bound

Each probe runs in a single-worker `ThreadPoolExecutor` whose `future.result(timeout)`
bounds the WHOLE probe call (default `PROBE_DEADLINE_MS = 8000`, matching the live
drivers' per-call deadline). A hung backend cannot hang the probe job or the worker —
the timeout records `failed` and the loop moves on. This is the EasyRoutes/Gadget
per-call ThreadPool pattern, applied ONCE in the probe runner rather than duplicated
per driver (gadget/easyroutes also self-bound internally; the outer wrapper is the
authoritative bound + belt-and-braces for the two that don't). Pinned by
`test_a_hung_check_hits_the_deadline_and_is_failed`.

## Result storage + retention

**Migration `0015_integration_probe.sql`** (verified free on this branch — max was
0014 — and on origin/main, which stops at 0011). Table:
`integration_probe(id, integration_key, status, reason, checked_at)` with a
`CHECK (status IN ('ok','failed','not_configured'))` and index
`(integration_key, checked_at DESC)` that serves both the latest-per-integration read
and the retention prune's range delete.

`record_probe_results()` inserts one cycle and prunes rows older than
`PROBE_RETENTION_DAYS = 30` in one transaction (the page only ever reads the latest
row per integration, so history depth is operator-forensics only). Two connection
modes like `PostgresJobQueue` — a fresh direct connection in production, an injected
one in tests. Unlike the fire-and-forget metric emit, a write failure RAISES so the
job retries/dead-letters rather than reporting success on a cycle the page never saw.
Pinned by `test_retention_prunes_probe_history_beyond_the_window` and
`test_record_fills_the_pages_last_probe_with_the_latest_per_integration`.

## How the page's `last_probe` now fills

The S15 handler left `conn` accepted-and-ignored as the deliberate seam. S16 adds
`_latest_probes(conn)` in the same `handlers/integrations.py`:
`SELECT DISTINCT ON (integration_key) … ORDER BY integration_key, checked_at DESC`,
and each row's `last_probe` is now `{status, reason, checked_at}` (or `None` →
"never probed") instead of a bare null. (`conn is None` only in the pure
config-presence unit tests, which have no probe history — they still see all-null,
unchanged.)

The shape widened from `last_probe: string|null` to an object, because the brief
requires a page **badge** and a badge needs the status. This rippled the honest,
minimal cost:
- **BFF** (`lib/bff/admin/integrations.ts`): new `ProbeResult` interface + strict
  `mapProbe` (a present-but-malformed probe throws 502, a null stays "never probed").
- **Panel** (`components/admin/IntegrationsPanel.tsx`): a `ProbeBadge` — green
  "Healthy" (ok), red "Failed" + reason (failed), muted "Skipped" (not_configured),
  plain "never probed" (null).
- **Mock stub** (`drivers/mock/admin_stubs.py`): unchanged — already `last_probe:
  None` (a pure-mock run has no probe table).

## Secret safety (NFR-6)

A `ProbeResult` carries a status + a reason STRING only; no credential value is ever
read into it, logged, or stored. Driver error messages reference env-derived URLs and
status codes, never a token. `node scripts/secret-scan.mjs --selfcheck && …` clean.

## Owner-blocked — what needs the owner's keys to go live

All seven credentials are owner-blocked, so a live probe records `not_configured` for
every row today. To go live, the owner sets (on the `background-worker` process,
which already inherits the docker-compose `env_file`):
`COMPOSIO_API_KEY` + `COMPOSIO_{SHOPIFY,QBO,SQUARE}_CONNECTED_ACCOUNT_ID` +
`COMPOSIO_TOOLKIT_VERSION_*`, `EASYROUTES_API_TOKEN` + `EASYROUTES_CLIENT_ID`,
`SIMPLETEXTING_API_TOKEN`, `OPENROUTER_API_KEY`, `GADGET_API_KEY`. At that point the
five UNVERIFIED wire details (Composio connected-account SDK surface, SimpleTexting
account path, OpenRouter `/key`, EasyRoutes deliveries endpoint, Gadget query) must be
confirmed against a live response — each fails closed to `failed` on a wrong guess, so
confirmation is safe to do incrementally. No compose change is required (the worker
shares the tool-executing env and already overrides `DATABASE_URL`).

## `ponytail:` ceilings left

1. `background_worker.INTEGRATION_PROBE_INTERVAL_SECONDS` — 15 min; shorten toward
   5 min if faster detection is worth the linear `job`-row growth (ADR-0153 fix #5).
2. `integration_probe.PROBE_DEADLINE_MS` — 8 s, matches the drivers; env-tune only if
   a backend needs a different budget.
3. `integration_probe.PROBE_RETENTION_DAYS` — 30 days; widen if an operator ever
   wants a longer probe timeline.
4. `integration_probe.record_probe_results` — fresh direct connection per cycle (like
   metric emit), not pooled; pool only if the cadence grows enough to matter.
5. `openrouter._KEY_PROBE_PATH` / the Composio/SimpleTexting/EasyRoutes/Gadget probe
   endpoints — UNVERIFIED live wire, confirm at cutover (fail closed on a wrong guess).
