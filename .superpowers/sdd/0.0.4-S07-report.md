## What moved and where

- **New file:** `packages/shared/src/memory.ts` — `MemoryPreferenceSlot` type
  + `MEMORY_PREFERENCE_SLOTS` const, moved verbatim (same four literals, same
  `readonly MemoryPreferenceSlot[]` shape).
- `packages/shared/src/index.ts` — added `export * from "./memory";` (matches
  the existing one-export-per-module barrel pattern already used for
  `profiles`/`routes`/`tools`/`channel-events`/`identity`).
- `packages/domain-adapters/src/mock/memory.ts` — no longer declares
  `MemoryPreferenceSlot`/`MEMORY_PREFERENCE_SLOTS` locally; imports both from
  `@toee/shared` (a package it already depends on — see
  `packages/domain-adapters/src/shared-contracts.test.ts`, which already
  exercises this exact "domain-adapters consumes @toee/shared contracts"
  pattern) and re-exports them so nothing else in the package's own barrel
  chain (`mock/index.ts` → `index.ts`) changes shape. This keeps the mock
  driver's own internal helpers (`isPreferenceSlot`, `requireSlot`,
  `CustomerPreferenceSlots`, `createMemoryMockHandlers`, etc. — all mock
  behavior, untouched) working unchanged.
- `apps/workbench/lib/gateway/types.ts` — import switched from
  `@toee/domain-adapters` to `@toee/shared`; two stale comments updated to
  stop naming domain-adapters as the source of truth.
- `apps/workbench/lib/gateway/types.test.ts` — import switched to
  `@toee/shared`; header comment, `describe`/`it` text updated to match (the
  test still asserts `PREFERENCE_SLOTS` is the *same array instance* as
  `MEMORY_PREFERENCE_SLOTS`, now sourced from `@toee/shared`).

Everything downstream of `apps/workbench/lib/gateway/types.ts` (
`hermes-agent-client.ts`, `hermes-map.ts`, `lib/bff/copilot/preferences.ts`,
`lib/api/admin-client.ts`, `lib/api/copilot-client.ts`) already imported
`MemoryPreferenceSlot` transitively through `./types` (a workbench-local
relative import), not directly from `@toee/domain-adapters` — so none of them
needed changes.

## Python-pairing note added

`packages/shared/src/memory.ts`'s header comment states Python's authoritative
copy lives at `hermes/toee_hermes/drivers/mock/memory.py::MEMORY_PREFERENCE_SLOTS`
and that both lists must be updated together. This matches the existing
`packages/shared/src/tools.ts` convention, which similarly footnotes the
Python-side file (`hermes/toee_hermes/tool_catalog.py`) inline near the
mirrored data, and the reverse direction already established in
`hermes/toee_hermes/tool_catalog.py`'s own docstring ("Ported from the TS
`@toee/shared` catalog so both runtimes share one source of truth.") and
`hermes/toee_hermes/drivers/mock/memory.py`'s own docstring ("ports
mock/memory.ts, ADR-0114"). No Python file was modified — the brief and the
verification block both scope Python as "should be untouched" for this slice,
and `MEMORY_PREFERENCE_SLOTS` in `hermes/toee_hermes/drivers/mock/memory.py`
was already independent of the TS mock's copy (no import, just a hand-mirrored
literal), so moving the TS copy doesn't orphan or invalidate anything on the
Python side.

## Concerns

- None blocking. `ToolExecutionContext` was the one genuinely borderline call:
  it's imported into `deps.ts` alongside `ToolDriver`, and in a mock-free world
  a "tool execution context" shape (profile + identity) would still plausibly
  exist as a contract. I left it with the mock-behavior group because the
  brief names `ToolDriver` (its sibling in the same dispatch layer) explicitly
  as staying, and `ToolExecutionContext` is defined in
  `packages/domain-adapters/src/tool-gate.ts` alongside Tool Gate enforcement
  logic that is unambiguously mock/dispatch-era plumbing, not a standalone
  data contract — moving just the type without its enforcement neighbor would
  be the "re-shaping the API while moving it" the brief warns against. Worth a
  second look once S09 is underway and its replacement shape is known.
