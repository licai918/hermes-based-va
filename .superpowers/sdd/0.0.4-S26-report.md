## 1. Live probe (phase 1 of the brief — done before anything was built)

Probed against the live Composio backend using the deployment `.env`, at the
pinned toolkit version **`square 20260616_00`**. Not documentation, not
assumption — three `get_raw_composio_tool_by_slug` resolutions and one real
`tools.execute`.

| Slug | Result at pin `20260616_00` |
|---|---|
| `SQUARE_RETRIEVE_PAYMENT_LINK` | **EXISTS** — `toolkit=square`, `version=20260616_00` |
| `SQUARE_CREATE_PAYMENT_LINK` | **404** `Tool_ToolNotFound` (confirms S12) |
| `SQUARE_LIST_PAYMENT_LINKS` | **404** `Tool_ToolNotFound` |

The Square toolkit carries **122 tools** at this pin. `SQUARE_RETRIEVE_PAYMENT_LINK`
is the only payment-link action among them — there is no list, no search, no
create.

### Actual parameter shape (live, not documented)

```json
{"type": "object", "title": "RetrievePaymentLinkRequest",
 "required": ["id"],
 "properties": {"id": {"type": "string", "examples": ["PAY_LINK_ID_123"],
   "description": "The Square-assigned ID of the payment link to retrieve."}}}
```

**One parameter. The Square-assigned link id. Nothing else.**

### Actual response shape (live, not documented)

`PaymentLink` carries `id`, `url`, `longUrl`, `orderId`, `version`, `createdAt`,
`updatedAt`, `description`, `paymentNote`, `checkoutOptions`, `prePopulatedData`.

**There is no money field.** The amount lives on the referenced *order*, not on
the link.

### Two live findings the schema does not tell you

1. **Composio calls a vendor error a success.** The live execute returned:

   ```json
   {"data": {"errors": [{"category": "AUTHENTICATION_ERROR",
                         "code": "INSUFFICIENT_SCOPES",
                         "detail": "...must authorize... ORDERS_READ"}],
             "payment_link": null},
    "error": null, "successful": true}
   ```

   `successful: true`, `error: null`, and `data` *is* a dict — so
   `_ComposioSdkClient.execute_action` passes it straight through. A naive
   response mapper would shape `payment_link: null` into an all-`None` "result"
   and hand it to a customer. Recorded in the driver comment; the mapper must fail
   closed on `data["errors"]` / a null link (ADR-0020).

2. **The live envelope is snake_case (`payment_link`); the published schema says
   camelCase (`paymentLink`).** Anyone writing the mapper from the schema alone
   would read the wrong key.

3. The connected account **lacks `ORDERS_READ`**, which the action requires. This
   is owner setup, listed in §7.

Probe is now permanent: `composio_smoke.py` phase 2 resolves gated-off slugs
instead of skipping them, so `SQUARE_RETRIEVE_PAYMENT_LINK` is re-verified on
every smoke run. Live output:

```
[PASS] toee_square_payment_link.send_payment_link -> SQUARE_RETRIEVE_PAYMENT_LINK (gated off in the driver)  -- toolkit=square
[PASS] toee_qbo_read.get_ar_summary -> QUICKBOOKS_GET_AGED_RECEIVABLES_REPORT (gated off in the driver)  -- toolkit=quickbooks
```

(The other gated-off spec is probed for free by the same change, and passes.)

## 2. The parameter contract — STOP, this is the owner's call

The brief said: *"If the only workable answer needs something the owner must set
up (e.g. a naming convention or a fixed link per purpose), STOP and report that
rather than inventing a shape."* **That condition fired.** Here is the reasoning
that got there, so the owner can overrule it with facts rather than trust.

**What identifies the link:** the Square-assigned payment link id. That is the
only input the action takes.

**Where could it legitimately come from?** Everything the agent holds at
`send_payment_link` time, checked one by one:

| Candidate | Verdict |
|---|---|
| Model-supplied param | **Forbidden.** ADR-0148: identity- and money-bearing values come from framework/tool context. A wrong id is not a failed lookup — it is a *successful* retrieve of a real link for someone else's amount, texted to a verified customer. This is the single worst failure available in this tool. |
| Session Identity Snapshot | Carries `shopify_customer_id`. No Square link. |
| QBO invoice read | Carries `DocNumber`, `Balance`, `BillEmail`. No Square link. |
| Conversation id | Thread identity only. |
| A Square console **naming convention** | **Does not work.** Resolving a name to an id needs a list or search action. `SQUARE_LIST_PAYMENT_LINKS` 404s at the pin. There is nothing to search with. |
| Owner-maintained configuration | **The only workable answer** — and its shape is a product decision (one fixed link for all payments vs. one link per invoice), not an engineering one. |

So the id can only come from owner config, which is exactly the STOP case. I did
not invent an env-var format, a map syntax, or a fixed-link convention.

**A second, independent blocker the owner also has to rule on:** the public
contract returns `amount`, and ADR-0066 requires Hermes to confirm the amount
before send. The retrieved `PaymentLink` has no money field. Recovering the amount
needs a second call (`SQUARE_RETRIEVE_ORDER`), which **ADR-0130 forbids** for one
v1 action. So either the owner accepts a send with no confirmed amount, or this
action needs the documented one-to-many exception under ADR-0130's "Future
exceptions".

**Consequence:** `ActionSpec.unavailable` stays. But the reason is now recorded
truthfully — the action exists and is named; what is missing is the link's
identity. The old fail-closed message ("the toolkit exposes no create-payment-link
action") was about to become false, which is exactly the class of stale claim this
iteration keeps getting burned by.

> Note for the owner: the S12 decision log recorded this switch as *"CLOSES the
> capability gap (production can send a link again)"*. The live probe says it does
> not, on its own. Retrieve is necessary but not sufficient; §7 is what makes it
> sufficient.

### `hermes/toee_hermes/drivers/mock/square.py` — mock parity

divergence the brief warned about.

- `SquareMockData.payment_link_base_url` + `payables` → `payment_links`.
- Unknown invoice → refused (`policy_blocked`), same as an unowned one: nothing to

## 4. Persona / tool description — exact wording

This is the highest-risk part: a description that still reads like "create"
produces an agent that confidently promises customers links that do not exist.
`hermes/toee_hermes/persona.py`, replacing the old two-line bullet
(`send_payment_link {invoice_number}` — sends on the customer's own verified
thread…):

```
- toee_square_payment_link — sends a payment link that ALREADY EXISTS. You cannot
create, generate, issue, or set up a payment link, and you cannot choose or change its
amount: every link is set up in advance by the team for one specific invoice, and this
tool only looks that link up and sends it. So NEVER tell a customer you will "create",
"generate", "make", or "set up" a link, and never say a link is on its way before the
tool has actually returned one.
  - `send_payment_link {invoice_number}` — sends the EXISTING link for that invoice on
the customer's own verified thread. NEVER include a `recipient` or any alternate
phone/address the customer typed in the message; redirecting a payment link is blocked.
If they ask to send it elsewhere, do NOT call this tool — open a follow-up case instead.
  - If this tool returns an error, there is no link for you to send — that includes the
case where no link has been set up for that invoice yet. Do NOT offer to make one and do
NOT invent a payment URL. Tell the customer you will have the team send the link, and
open a follow-up case.
```

**Why it cannot be read as "create":**

1. The capability sentence leads with what the tool *is* — "sends a payment link
   that **ALREADY EXISTS**" — before naming the action, so the frame is set before
   the model reads `send_payment_link`.
2. It states the negative **explicitly and in the customer's vocabulary**, not
   ours: *create, generate, issue, set up, make*. A model that avoids only the word
   "create" will still say "I'll generate one for you"; all five verbs are closed.
3. It separates **doing** from **saying**. "Never *tell a customer* you will…" —
   the failure mode is a promise in the reply text, not just a wrong tool call, and
   promises are what reach the customer.
4. It removes the amount as an agent choice ("cannot choose or change its
   amount"), which is the money-bearing half of ADR-0148.
5. It closes the **timing** hole: "never say a link is on its way before the tool
   has actually returned one" — the agent cannot announce first and discover later.
6. It names the **empty case** as an error, not a gap to fill: "there is no link
   for you to send… Do NOT offer to make one and do NOT invent a payment URL."
   Without this the model reads a failed lookup as an invitation to improvise, and
   the existing "never fabricate" section does not obviously cover *URLs*.

The persona is imported by both `hermes_runtime.eval_record` and
`make_openrouter_run_turn`, so recorded and live behavior share this wording.

## 5. ADR-0130 resolution note

The S12 **Correction** note is left intact (it is the record of a probed fact); a
new **Resolution (0.0.4 S26, 2026-07-22)** note follows it, and the mapping table's
Square row changed from *"one Square payment-link toolkit action — does not exist"*
to *"one Square payment-link **retrieve** toolkit action — mapped but still gated"*.

The note records: the owner chose retrieve semantics; the live probe result and
the exact parameter shape; that **ADR-0130's rule needs no escape hatch and no
exception for the retrieve itself** (it is one call for one action); and the two
things that keep it gated — link identity (ADR-0148) and the amount, where the
one-call rule is the actual constraint and ADR-0130's own "Future exceptions"
section is the path if the owner wants a confirmed amount.

Runbook §6.5 rewritten to match, and turned from a "known gap" narrative into the
three-item owner setup checklist reproduced in §7.

No new ADR number was needed — this is a resolution note on an existing ADR — so
the 0155/0156-vs-main collision risk was not touched.

## 7. Owner setup needed in Square before this can work in production

1. **Decide how a link is identified** — one fixed link for all payments, or one
   pre-created link per invoice. This decides the config shape, which is why it was
   not invented here.
2. **Supply the Square-assigned link *ids*, not the console URLs.** The retrieve
   action takes an id. A naming convention in the console cannot be resolved,
   because the toolkit has no list/search action at the pin.
3. **Re-authorize the Square connected account with `ORDERS_READ`.** The live
   execute failed `INSUFFICIENT_SCOPES` on exactly this scope — and Composio
   reported that failure as `successful: true`, so it will not announce itself.
4. **Rule on the amount.** ADR-0066 requires a confirmed amount before send; a
   retrieved link has none. Either drop the confirmation, or accept an ADR-0130
   "Future exceptions" entry for a second call.
5. **Pre-create the actual links** for the invoices customers will be sent (at
   minimum INV-9001 if the eval suite is ever run against live Square).

Once 1–4 land, the code change is small and mechanical: request mapper →
`{"id": <from config>}`, response mapper → `payment_link.url` with a fail-closed
check on `data["errors"]` / null link, and delete `unavailable`.

## 9. Concerns

- **The gap is not closed, contrary to the decision log's expectation.** Retrieve
  exists, but with no governed link id the tool is still fail-closed in production.
  §7 is the remaining distance.
- **The amount problem is structural, not incidental.** ADR-0066 (confirm the
  amount before send) and ADR-0130 (one Composio call per v1 action) cannot both
  hold for a retrieved payment link. One of them has to give, and that is an owner
  call.
- **`successful: true` on a vendor error is a live fail-open shape.** It bit
  nothing here because the action is gated, but it is generic to
  `_ComposioSdkClient` — any Composio action whose vendor returns a structured
  error inside `data` reaches the response mapper as a "success". Only Square's
  retrieve is documented against it so far; whether other toolkits do this is
  unprobed.
- **The mock still returns `amount`** and production may not be able to. Flagged in
  the mock docstring; resolves with owner item 4.
