# 0.0.4 S09 — API-only cutover: delete the in-memory fallback

**Status:** DONE
**Branch:** `feat/0.0.4-land-all`
**Commits:** `cda2b17` (code), `61016f5` (docs)
**Delivers:** FR-1, FR-3, FR-4, NFR-4

The workbench carried two backends since Slice 2. It now carries one. Every BFF
route reads and writes through the per-profile Hermes API; a missing
`HERMES_*_API_URL/TOKEN` stops the server at boot instead of silently swapping in
a store that forgets everything on reload.

## 3. Fail-closed at boot, and its exact message

`resolveProfileApiConfig(url, token) → ProfileApiConfig | null` is replaced by two
functions over an injectable env, in `lib/gateway/hermes-api-config.ts`:

- `requireProfileApiConfig("copilot" | "admin", env?)` — throws naming that
  profile's missing variables;
- `assertHermesApiConfig(env?)` — the boot gate; reports **all** missing variables
  in one message rather than one per restart.

`apps/workbench/instrumentation.ts` calls it from Next's `register()` hook. It is
guarded to the Node runtime (the edge/middleware runtime never talks to the API)
and skipped during `phase-production-build` (so `next build` does not need
production credentials — CI runs `pnpm -r typecheck` + `pnpm test`, never
`next build`).

**Verified live**, not just unit-tested. With two of the four variables removed
from `.env.local`:

## 4. The four traps, and what happened to each

1. **`lib/auth/password.ts` survives.** Only `verifyPassword` was removed;
   `hashPassword` + `validatePassword` stay because `lib/bff/admin/accounts.ts`
   hashes admin-created passwords in the Next.js **server** runtime and sends only
   the hash (ADR-0144). The module header now says so explicitly, names the scrypt
   parameters that must not drift, and points at hermes-runtime's
   `TS_GENERATED_HASH` test as the other half of the contract. `password.test.ts`
   keeps its format test for the same reason.
2. **`handleSession`'s unused `store`.** Dropped the field (`AuthDeps` →
   `SessionDeps = { now, secret }`); the route stays. `app/api/auth/session/route.ts`
   now makes **no backend call at all** — it reads the signed cookie and nothing
   else, which is worth knowing when reasoning about workbench availability.
3. **`docs/ops/local-e2e.md`.** Fixed — see §6.
4. **`ToolExecutionContext`.** Conclusion below.

## 5. `ToolExecutionContext` — the S07 judgment call, resolved for S11

**Verdict: leave it in `packages/domain-adapters` and let S11 delete it with the
package. It does not need to move to `packages/shared`.**

S07 left it with the mock-behavior group but flagged it as the closest call, worth
a second look "once S09 is underway and its replacement shape is known". The
replacement shape is now known: there isn't one.

`ToolExecutionContext` reached the workbench through exactly one call site —
`copilotContext(deps)` in `lib/bff/copilot/deps.ts`, feeding `executeTool`. Both
are deleted. In the API-only world the equivalent notion (which profile, which
acting employee) is carried on the wire by `HermesApiClient`'s `actorAccountId`
and by the *server the token points at* — the profile is a property of the
dispatch server, not a parameter the client passes. There is no TS shape left for
`ToolExecutionContext` to be the contract *of*.

Grep proof (whole repo, excluding `packages/domain-adapters` itself):

```
$ grep -rn "ToolExecutionContext" apps packages/shared packages/eval-runner eval scripts services
packages/eval-runner/src/harness.ts:17:  type ToolExecutionContext,
packages/eval-runner/src/harness.ts:125:): ToolExecutionContext {
```

The only remaining consumer is the TS `packages/eval-runner`, which S11 deletes
too. So S11 can delete `ToolExecutionContext` cleanly along with `tool-gate.ts`,
`execute-tool.ts` and the rest — no extraction to `@toee/shared` required.
